#### lib/analysis/heat.ts 개선

**추가된 함수**:

1. **`calculateHeatIndex()` - UI 표시용 시간 가중 화력**

   - 시간 가중치 적용 (7일 100% → 30일 10%)

```typescript
function getTimeWeight(createdAt: string): number {
    const age = Date.now() - new Date(createdAt).getTime();
    const daysSinceCreated = age / (1000 * 60 * 60 * 24);
    
    if (daysSinceCreated <= 7) return 1.0;      // 100%
    if (daysSinceCreated >= 30) return 0.1;     // 10%
    return 1.0 - ((daysSinceCreated - 7) / 23) * 0.9;  // 선형 감소
}
```

2. **`calculateRecentHeat()` - 상태 전환용 최근 화력**
   - 최근 7일 데이터만 사용 (시간 필터)
   - 시간 가중치 없음
   - 기존 임계값 유지 (30점/10점)

## 개선 방안

### 방안 1: 시간 가중치 기반 화력 계산 (권장)

#### 개념

화력을 "누적 영향력"이 아닌 **"시간 가중 영향력"**으로 변경

- 최근 데이터에 높은 가중치
- 오래된 데이터에 낮은 가중치
- 자연스러운 화력 감소

#### 구현

```typescript
// lib/analysis/heat.ts

export async function calculateHeatIndex(issueId: string): Promise<number> {
    const now = Date.now()
    
    // 시간 제한 없이 모든 데이터 조회 (created_at 필요)
    const [communityResult, newsResult] = await Promise.all([
        supabaseAdmin
            .from('community_data')
            .select('view_count, comment_count, created_at')
            .eq('issue_id', issueId),
        supabaseAdmin
            .from('news_data')
            .select('source, published_at, created_at')  // published_at 또는 created_at
            .eq('issue_id', issueId),
    ])

    const communityData = communityResult.data ?? []
    const newsData = newsResult.data ?? []

    // 시간 가중치 함수
    function getTimeWeight(dateStr: string): number {
        const age = now - new Date(dateStr).getTime()
        const daysSincePublished = age / (1000 * 60 * 60 * 24)
        
        // 옵션 A: 지수 감소 (급격한 감소)
        // 7일 후 50%, 14일 후 25%
        // return Math.pow(0.5, daysSincePublished / 7)
        
        // 옵션 B: 선형 감소 (완만한 감소) - 권장
        // 7일까지 100%, 이후 서서히 감소, 30일 후 10%
        if (daysSincePublished <= 7) return 1.0
        if (daysSincePublished >= 30) return 0.1
        return 1.0 - ((daysSincePublished - 7) / 23) * 0.9
        
        // 옵션 C: 단계별 감소 (명확한 구간)
        // if (daysSincePublished <= 3) return 1.0   // 최근 3일: 100%
        // if (daysSincePublished <= 7) return 0.8   // 1주일: 80%
        // if (daysSincePublished <= 14) return 0.5  // 2주일: 50%
        // if (daysSincePublished <= 30) return 0.2  // 1개월: 20%
        // return 0.05                               // 그 이상: 5%
    }

    // 커뮤니티 화력 (시간 가중)
    let communityHeat = 0
    if (communityData.length > 0) {
        let weightedViews = 0
        let weightedComments = 0
        let totalWeight = 0
        
        for (const item of communityData) {
            const weight = getTimeWeight(item.created_at)
            weightedViews += (item.view_count ?? 0) * weight
            weightedComments += (item.comment_count ?? 0) * weight
            totalWeight += weight
        }
        
        // 정규화 (평균 가중치로 나눔)
        const avgWeight = totalWeight / communityData.length
        const effectiveViews = weightedViews / avgWeight
        const effectiveComments = weightedComments / avgWeight
        
        const viewScore = Math.min(100, (effectiveViews / 5000) * 100)
        const commentScore = Math.min(100, (effectiveComments / 500) * 100)
        const raw = viewScore * 0.35 + commentScore * 0.45
        communityHeat = Math.min(100, Math.max(0, Math.round(raw)))
    }

    // 뉴스 신뢰도 (시간 가중)
    let newsCredibility = 0
    if (newsData.length > 0) {
        let weightedCount = 0
        const weightedSources = new Map<string, number>()
        
        for (const item of newsData) {
            const weight = getTimeWeight(item.published_at || item.created_at)
            weightedCount += weight
            
            const currentWeight = weightedSources.get(item.source) || 0
            weightedSources.set(item.source, currentWeight + weight)
        }
        
        // 출처 다양성 (가중치 기반)
        const effectiveSources = Array.from(weightedSources.values())
            .reduce((sum, w) => sum + Math.min(1, w), 0)
        
        const sourceScore = (Math.min(20, effectiveSources) / 20) * 100
        const countScore = Math.min(100, weightedCount * 2)
        newsCredibility = Math.min(100, Math.max(0, Math.round(sourceScore * 0.6 + countScore * 0.4)))
    }

    // 기존 화력 공식 동일
    const communityAmp = communityHeat <= 3
        ? 0
        : Math.min(1, Math.sqrt(Math.max(0, communityHeat - 3) / 70))

    const heatIndex = Math.round(
        Math.min(100, Math.max(0, newsCredibility * (0.3 + 0.7 * communityAmp)))
    )

    await supabaseAdmin
        .from('issues')
        .update({
            heat_index: heatIndex,
            updated_at: new Date().toISOString(),
        })
        .eq('id', issueId)

    return heatIndex
}
```

## 심각한 오류 발견

### 오류 1: 상태 전환 로직과의 치명적 모순

#### 문제 분석

현재 상태 전환은 **화력 임계값**에 의존:

```typescript
// lib/analysis/status-transition.ts
점화 → 논란중: heat >= 30점
논란중 → 종결: heat < 10점
종결 → 논란중: heat >= 30점 AND 최근 48시간 데이터 있음
```

시간 가중 화력 적용 시:
- **지속적 발생 이슈**: 매일 뉴스 1-2건 (10일) → 시간 가중으로 20점
- **결과**: 30점 미달로 "논란중" 전환 실패
- **모순**: 활발한 이슈인데 "점화" 고착

더 심각한 문제 - 재점화 불가:
- 종결 이슈에 신규 뉴스 5건 발생
- 과거 데이터는 가중치 0.1 → 화력 15점
- 조건: `heat >= 30` 미충족
- **재점화 로직 사실상 비활성화**

#### 해결 방안

**Option A: 화력 임계값 하향 (단순, 위험)**
```bash
STATUS_IGNITE_MIN_HEAT=15  # 30 → 15
STATUS_CLOSED_MAX_HEAT=3   # 10 → 3
```
장점: 간단
단점: 저품질 이슈 노출 위험

**Option B: 상태 전환용 별도 화력 (권장)**
```typescript
// 상태 전환: 최근 7일 화력 (시간 필터)
const recentHeat = await calculateRecentHeat(issueId, 7)

// UI 표시: 시간 가중 화력 (전체 데이터)
const displayHeat = await calculateHeatIndex(issueId)

if (recentHeat >= 30) {
    newStatus = '논란중'
}
```
장점: 명확한 역할 분리, DB 변경 없음
단점: 화력 계산 2회

**Option C: 이중 화력 컬럼 (복잡, 완벽)**
```sql
ALTER TABLE issues ADD COLUMN cumulative_heat INTEGER;
-- heat_index: 시간 가중 (UI)
-- cumulative_heat: 누적 (상태 전환)
```
장점: 완벽한 분리
단점: DB 마이그레이션 필요

### 오류 2: 뉴스 발행 시점 데이터 부재

#### 문제

제안 코드:
```typescript
const weight = getTimeWeight(item.published_at || item.created_at)
```

**현실**:
- `news_data`에 `published_at` 컬럼 없음
- 현재 스키마: `id, issue_id, source, title, url, created_at`
- 뉴스 발행 시점이 아닌 수집 시점만 기록

**시나리오**:
```
2026-03-01: 뉴스 A 발행
2026-03-10: 크롤러가 뉴스 A 발견 (created_at: 2026-03-10)
가중치 계산: 1일 전 → 1.0
실제: 10일 전 뉴스 → 0.43
오차: 2.3배
```

#### 해결 방안

**Option A: created_at 사용 (현실적)**
```typescript
const weight = getTimeWeight(item.created_at)
```
장점: 즉시 적용 가능
단점: 부정확 (수집 시점 기준)

**Option B: published_at 추가 (정확)**
```sql
ALTER TABLE news_data ADD COLUMN published_at TIMESTAMP;
CREATE INDEX idx_news_published ON news_data(published_at);
```
```typescript
// 네이버 API pubDate 파싱
const publishedAt = new Date(item.pubDate).toISOString()
```
장점: 정확한 시간 가중치
단점: DB 마이그레이션, 크롤러 수정, 기존 데이터 null

**Option C: metadata JSON 컬럼**
```sql
ALTER TABLE news_data ADD COLUMN metadata JSONB;
```
```typescript
metadata: { pubDate: item.pubDate, source_reliability: 0.9 }
```
장점: 유연성
단점: 복잡도 증가

### 오류 3: 가중치 정규화 수학 오류 (Minor)

#### 문제

제안 코드:
```typescript
const avgWeight = totalWeight / communityData.length
const effectiveViews = weightedViews / avgWeight
```

**수학적 오류**:
```
effectiveViews = weightedViews / (totalWeight / count)
              = weightedViews × count / totalWeight
```

**예시**:
```
데이터: [100조회, 100조회, 100조회]
가중치: [1.0, 0.5, 0.1]
weightedViews = 160
avgWeight = 0.533
effectiveViews = 300  ← 오히려 증가!
```

#### 해결

정규화 제거:
```typescript
let weightedViews = 0
for (const item of communityData) {
    const weight = getTimeWeight(item.created_at)
    weightedViews += (item.view_count ?? 0) * weight
}
const viewScore = Math.min(100, (weightedViews / 5000) * 100)
```

## 추가 고려사항

### 상태 전환 로직 영향

현재 상태 전환 로직은 화력 임계값 사용:
- 점화 → 논란중: 화력 30점 이상
- 논란중 → 종결: 화력 10점 미만

**시간 가중 화력 도입 시**:
- 자연스럽게 화력이 감소하므로 상태 전환도 자연스러워짐
- "최근 48시간 신규 수집 없음" 조건과 일관성 증가
- 별도 수정 불필요

### 화력순 정렬 개선

```typescript
// 이슈 목록 정렬
```
