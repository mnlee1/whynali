# 카테고리 시스템 확장 가이드

왜난리 프로젝트의 카테고리 시스템은 **확장 가능한 설계**로 개선되었습니다.
새로운 카테고리 추가 시 한 곳만 수정하면 전체 시스템에 자동 반영됩니다.

## 설계 개요

### 주요 개선사항
- **단일 진실 공급원**: `lib/config/categories.ts` 파일에서 모든 카테고리 설정 관리
- **동적 타입 생성**: TypeScript Union 타입이 설정 파일에서 자동 생성
- **DB 제약 제거**: CHECK 제약 삭제로 마이그레이션 없이 확장 가능
- **UI 자동 반영**: 색상/스타일이 설정 파일에서 자동 적용

### 아키텍처
```
lib/config/categories.ts (단일 진실 공급원)
    ↓
┌───┴───────────────────────────────────┐
│                                       │
types/issue.ts              UI Components
(타입 자동 생성)            (스타일 자동 적용)
    ↓                                   ↓
API/Backend                 CategoryBadge
(유효성 검증)               HotIssueHighlight
```

## 새 카테고리 추가 방법

### 1단계: 설정 파일 수정

`lib/config/categories.ts` 파일의 `CATEGORIES` 배열에 새 항목 추가:

```typescript
export const CATEGORIES: CategoryConfig[] = [
    // ... 기존 카테고리들 ...
    
    {
        id: '경제',              // 카테고리 ID (DB 저장값)
        label: '경제',           // 표시명
        badgeColors: {
            bg: 'bg-yellow-100',
            text: 'text-yellow-700',
            border: 'border-yellow-200',
        },
        gradientColors: 'from-yellow-500 via-orange-500 to-red-500',
        keywords: [
            '주식', '코스피', '환율', '금리', '증시', '달러',
            '경제', '재계', '기업', '투자', '수출', '수입',
        ],
        contextRules: [
            { keywords: ['주식', '폭락'], boost: 15 },
            { keywords: ['환율', '급등'], boost: 12 },
        ],
    },
]
```

### 2단계: DB 마이그레이션 실행

```bash
# Supabase CLI 사용
npx supabase db push

# 또는 SQL 직접 실행 (이미 적용됨)
psql -f supabase/migrations/remove_category_check_constraint.sql
```

**중요**: DB CHECK 제약 제거 마이그레이션은 이미 작성되어 있습니다.
최초 1회만 실행하면 이후 카테고리 추가 시 마이그레이션 불필요합니다.

### 3단계: 완료

설정 파일만 수정하면 자동으로 다음 항목들이 업데이트됩니다:
- TypeScript 타입 (`IssueCategory`)
- API 유효성 검증 (`VALID_CATEGORIES`)
- UI 컴포넌트 색상 매핑
- AI 키워드 분류 로직

## 설정 항목 설명

### CategoryConfig 인터페이스

```typescript
interface CategoryConfig {
    id: string                      // 카테고리 고유 ID (DB 저장값)
    label: string                   // 사용자에게 표시되는 이름
    badgeColors: {                  // 배지 컴포넌트 색상
        bg: string                  // 배경색 (Tailwind class)
        text: string                // 텍스트색
        border: string              // 테두리색
    }
    gradientColors: string          // 히어로 섹션 그라디언트
    keywords: string[]              // AI 분류용 키워드 목록
    contextRules: Array<{           // 맥락 기반 분류 규칙
        keywords: string[]          // 동시 등장 시 매칭되는 키워드 조합
        boost: number               // 매칭 시 추가 점수 (10-20 권장)
    }>
}
```

### 색상 선택 가이드

Tailwind CSS 색상 팔레트 사용 권장:
- 배경: `-100` (연한 배경)
- 텍스트: `-700` (진한 텍스트)
- 테두리: `-200` (중간 테두리)

그라디언트: `from-{color}-500 via-{color}-500 to-{color}-500` 형식

### 키워드 작성 가이드

1. **고유성**: 해당 카테고리를 명확히 구분하는 단어 선택
2. **다양성**: 다양한 표현 방식 포함 (한글/영어/약어)
3. **빈도**: 실제 뉴스에서 자주 등장하는 단어 우선
4. **중복 방지**: 다른 카테고리와 겹치는 범용 단어 제외

**예시**:
```typescript
// 좋은 예
keywords: ['야구', '축구', '선수', '감독', '경기', '우승']

// 나쁜 예 (너무 범용적)
keywords: ['사람', '문제', '상황', '발표']
```

### Context Rules 작성 가이드

특정 키워드 조합이 함께 등장하면 카테고리 점수를 크게 증가시킵니다.

```typescript
contextRules: [
    // 타이어 + OE → 기술 카테고리 확정 (자동차 산업 이슈)
    { keywords: ['타이어', 'OE'], boost: 15 },
    
    // 국회 + 법안 → 정치 카테고리 확정 (입법 이슈)
    { keywords: ['국회', '법안'], boost: 15 },
]
```

## 주의사항

1. **DB 데이터 정합성**: 기존 이슈의 카테고리는 자동으로 변경되지 않습니다
2. **캐시 클리어**: 프로덕션 배포 시 Next.js 캐시 클리어 필요
3. **키워드 중복**: 여러 카테고리에 같은 키워드가 있으면 맥락 규칙으로 구분
4. **타입 체크**: TypeScript 컴파일 에러 발생 시 `npm run build` 확인

## 롤백 방법

설정 파일에서 해당 카테고리 항목 제거 후 재배포하면 됩니다.
단, DB에 이미 저장된 해당 카테고리 데이터는 수동 정리 필요합니다.

```sql
-- 특정 카테고리 이슈 조회
SELECT * FROM issues WHERE category = '경제';

-- 다른 카테고리로 일괄 변경 (선택)
UPDATE issues SET category = '사회' WHERE category = '경제';
```
