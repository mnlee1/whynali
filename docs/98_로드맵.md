## 2. 서비스 개요

**한 줄 요약**

- 한국 이슈(사건/논란)를 한곳에 모아 시간순(발단→전개→파생→진정)으로 보여주는 웹 서비스
- 사용자가 감정 표현·댓글·투표로 반응 가능

**배경**

- 이슈 정보가 뉴스/커뮤니티/SNS에 분산됨
- "지금 무슨 일이 벌어졌고 왜 커졌는지" 한 번에 파악 어려움
- 발단→전개→파생→진정 시계열 정리 서비스에 대한 잠재 수요 존재

**목적**

- 이슈 흐름·여론 변화를 시간축으로 정리
- "현재 상황"과 "왜 이렇게 되었는지" 동시 제공
- 정보 탐색 비용 감소, 댓글·리액션으로 재방문 유도

**방향**

- **이슈 내비게이션**형: 이벤트 단위 타임라인·이슈 포인트 제공
- 모바일 퍼스트·반응형 웹
- 로그인: 댓글·감정·투표·토론에만 필요 / 이슈·타임라인 조회는 비로그인 가능

## 3. 메뉴 구조

역할 분담·기능 분할 시 이 구조를 기준으로 한다.

**사용자(공개·비로그인 가능)**

- 상단 메뉴: 네이버 뉴스 카테고리를 메뉴로 사용 (01·02 확정).
- **글로벌 검색**: 전체 메뉴에서 노출. 이슈 + 커뮤니티(토론 주제) 통합 검색.

```
[로고/홈]  [연예] [스포츠] [정치] [사회] [기술]  [커뮤니티]  [검색]  [로그인]
    |         \___________________/          |          |         |
    |                  |                    |          |         +-- 오픈ID
    |                  |                    |          +-- 글로벌 검색 (이슈 + 토론 주제 통합)
    |                  |                    |
    |                  |                    +-- 토론 주제 목록 (검색/필터 가능)
    |                  |                              |
    |                  |                              +-- 토론 주제 상세 (댓글, "해당 이슈 보기" 링크)
    |                  |
    |                  +-- 각 카테고리 = 이슈 목록(해당 카테고리)
    |                           |
    |                           +-- 검색·필터 (키워드 검색, 상태별: 점화|논란중|종결, 정렬)
    |                           |
    |                           +-- 이슈 상세
    |                                    |
    |                                    +-- 화력 분석(숫자+등급), 상태(점화/논란중/종결)
    |                                    +-- 타임라인(발단→전개→파생→진정, 출처 링크)
    |                                    +-- 감정 표현 7종 (로그인 시)
    |                                    +-- 댓글 (베스트 상단 + 리스트, 로그인 시)
    |                                    +-- 투표 (복수·시점별, 로그인 시)
    |                                    +-- "이 이슈의 커뮤니티" 링크 → 해당 이슈 토론 주제 목록
    |
    +-- 메인(홈) = 이슈 목록 전체(카테고리 무관) 또는 연예 등 기본 카테고리 리다이렉트
```

**검색·필터 정리**

| 구분 | 범위 | 기능 |
|------|------|------|
| 글로벌 검색 | 전체 서비스 | 이슈 + 커뮤니티(토론 주제) 통합 검색. 상단 검색창에서 입력 시 두 타입 모두 검색·노출. |
| 이슈 페이지 내 | 이슈 목록(해당 카테고리 또는 전체) | 키워드 검색, 상태별 필터(점화/논란중/종결), 정렬(최신·화력 등). |
| 커뮤니티 페이지 내 | 토론 주제 목록 | 키워드 검색, 필터(연결 이슈·카테고리 등). |

**관리자(전용·로그인 후 역할 구분)**

```
[관리자 진입]
    |
    +-- 이슈·타임라인 관리 (CRUD)
    +-- 수집 데이터
    |       +-- 뉴스 수집 목록 (제목·링크·출처·날짜), 이슈 연결
    |       +-- 커뮤니티 메타데이터 목록 (더쿠·네이트판), 이슈 연결
    +-- 화력 분석 (확인·승인 선택, 02에 따라 자동 반영 가능)
    +-- AI 토론 주제 후보 (승인/반려 후 커뮤니티에 등록)
    +-- 세이프티봇 검토 대기 (댓글·토론글 공개/삭제)
```

| 홈 / 연예·스포츠·정치·사회·기술 | 이슈 목록 API. 카테고리=연예|스포츠|정치|사회|기술(네이버 뉴스 기준), 검색·상태 필터·정렬 |

| 이슈 상세 – 투표 | 투표 API(목록·제출·결과), 복수·상태별 |
| 이 이슈의 커뮤니티 | 토론 주제 목록 API(이슈 ID 기준) |
| 커뮤니티 메뉴 | 토론 주제 목록 API(전체), 검색·필터 파라미터 |
| 커뮤니티(페이지 내 검색/필터) | 토론 주제 목록 API(q=키워드, 이슈·카테고리 등 필터) |
| 토론 주제 상세 | 토론 주제 상세 API, 토론 댓글 API(목록·작성), 세이프티봇 |
| 로그인 | Supabase Auth(구글·네이버·카카오) |
| 관리자 – 이슈/타임라인 | 이슈·타임라인 CRUD API |
| 관리자 – 수집 | 뉴스 수집 목록·커뮤니티 수집 목록 API, 이슈 연결 API |

## 5. 역할 분담 (2명)

**원칙**

- **처음부터 나눠서 작업하지 않는다.** 먼저 **기초·공통 부분을 함께 픽스**하고, 그 내용을 `dev`에 머지한 뒤 **`git pull` 받아서** 그 위에서만 A/B가 분할 작업한다.
- 분할 시에는 백엔드/프론트를 사람별로 나누지 않고, **담당 영역의 백엔드 + 프론트**를 한 사람이 맡는다(§3 메뉴 구조 기준).
- Git: 기초 단계는 한 브랜치에서 함께 정리·머지. 분할 단계부터는 `dev` pull → `feature/본인-기능명` → 작업 → PR(dev) → 머지. **99_Git협업.md** 참고.

---

**1단계: 기초·공통 픽스 (분할 전, 둘 다 합의 후 픽스)**

- 아래를 **한꺼번에 나누지 말고** 함께(또는 한 명이 초안 → 다른 한 명 검토) 진행하고, **dev에 머지까지 끝낸 뒤** 2단계로 넘어간다.
- 2단계에서 작업 시작 전에는 반드시 `git pull origin dev`로 이 기초 버전을 받은 상태에서 브랜치 생성.

| 구분 | 픽스할 내용 |
|------|-------------|
| **환경** | 저장소 생성, Next.js 프로젝트 생성, 기술 스택·로컬 실행 방법 문서, 환경 변수 템플릿(.env.example). |
| **기획·메뉴** | 기능 목록·DB에 넣을 데이터 목록·§3 메뉴 구조 합의. "어떤 화면이 어떤 데이터를 쓸지" 목록 확정. |
| **DB 스키마** | 테이블 목록·테이블별 컬럼·타입·외래키. Supabase에 테이블 생성까지. (아래 "DB 합의 시 관점" 참고.) 본문·요약문·수집 닉네임 컬럼 없음. |
| **API 규약·공통 타입** | 이슈 목록/상세 응답 형태, 댓글·투표·토론 주제 목록 요청/응답 예시를 문서로 고정. 저장소에 [97_API규약.md](./97_API규약.md)로 보관. |
| **앱 뼈대(선택)** | Next.js 라우팅 구조(홈, /연예·스포츠·정치·사회·기술, /커뮤니티, /이슈/[id] 등). 공통 레이아웃(헤더·메뉴·검색 자리). 이렇게 픽스해 두면 분할 후 A/B가 각자 해당 경로·컴포넌트만 채움. |

**2단계: 분할 작업 (기초 픽스 머지 후, pull 받고 진행)**

- `git checkout dev` → `git pull origin dev` 로 기초 버전 받기 → `git checkout -b feature/본인-기능명` 후 작업.
- 아래 담당 A/B는 **1단계가 끝난 뒤**에만 적용.

**DB 합의 시 맞춰야 할 관점** (1단계 스키마 설계 시 사용)

- **테이블 목록·역할**: issues, timeline_points, comments, reactions, votes, vote_choices, discussion_topics, news_data, community_data, users, safety_rules 등(02 §8.3 참고).
- **테이블별 필드·타입**: 컬럼명·타입(id, uuid, text, int, timestamp 등). **외래키**(issue_id 등) 이름·참조 테이블 통일.
- **저장하지 않기로 한 것**: 본문·요약문(description)·닉네임(수집) 컬럼 없음.
- **이름 규칙**: 테이블명·컬럼명(스네이크 vs 캐멀), id 규칙. 합의 결과는 스키마 문서로 저장소에 보관.

**이슈 상세 페이지 협업**

- **A**: 상단 레이아웃, 화력·상태, 타임라인·출처 링크
- **B**: 감정·댓글·투표 블록, "이 이슈의 커뮤니티" 링크
- 한 페이지이므로 컴포넌트 단위로 나누고, 합칠 때 A 또는 B가 레이아웃에 B 컴포넌트 임포트. 충돌 줄이려면 `app/`, `components/` 아래에서 담당 디렉터리·파일을 나눌 것.

**2단계 담당 범위 검증** (§3 메뉴·화면 기준, 겹침·공백 없음)

| 메뉴/화면 | 백엔드 담당 | 프론트 담당 |
|-----------|-------------|-------------|
| 글로벌 검색 | B (이슈+토론 통합 검색 API) | B (검색 UI·결과) |
| 홈, 연예·스포츠·정치·사회·기술, 이슈 목록(검색/필터) | A (이슈 목록 API) | A |
| 이슈 상세 (화력·타임라인·출처) | A (이슈 상세·타임라인·출처 목록 API) | A |
| 이슈 상세 (감정·댓글·투표) | B (감정·댓글·투표 API, 세이프티) | B |
| "이 이슈의 커뮤니티" 링크, 커뮤니티 메뉴·토론 목록/상세 | B (토론 주제·토론 댓글 API) | B |
| 로그인 | B (Auth) | B |
| 관리자 – 이슈/타임라인/수집/화력 | A | A |
| 관리자 – 토론 주제 승인·세이프티 검토 | B | B |

- 수집(뉴스·커뮤니티) 저장·이슈 연결·화력 분석: 전부 A. B는 수집 데이터를 직접 쓰지 않고, 토론 주제·댓글·세이프티만 담당.
- 글로벌 검색 API는 B가 이슈·토론 주제 테이블 모두 조회(읽기 전용)하여 통합 검색. 이슈 목록 API(A)와 별개.

## 7. 옵션 정의 (구현 시 기준)

- **이슈 카테고리**: 연예, 스포츠, 정치, 사회, 기술. 네이버 뉴스 API 카테고리와 1:1 매핑.
- **이슈 상태**: 점화(Ignited), 논란 중(Debating), 종결(Closed). 표시: 텍스트 + 아이콘(02 확정).
- **로그인**: 구글·네이버·카카오. 메타는 확장 시 검토.
- **감정 표현**: 좋아요, 싫어요, 화나요, 팝콘각, 응원, 애도, 사이다. 전 옵션 구현. 이슈당 1인 1회, 변경·취소 가능.
- **댓글 정렬**: 상단 베스트(score = 좋아요−싫어요 상위) / 아래 리스트: 최신순·좋아요순·싫어요순 중 선택 (02 6.4)
- **투표**: 이슈당 복수 투표(시점/상황별), 선택지 2~4개, 실시간 그래프/비율, 이슈 상태별 이전 투표 노출 (02 6.4)
- **타임라인**: 위→아래 시간순(발단→전개→파생→진정), 카드 형태 / 상단 화력 분석, 하단 출처 링크
- **뉴스 수집**: 네이버 뉴스 API
  - 제목·링크·출처·날짜만 수집·저장·표시 (요약문 미수집·미표시, 04 4.3.1)
  - 30분 주기, 한도 80% 알림·주기 완화
- **커뮤니티 수집**: 더쿠·네이트판 2곳
  - 메타데이터만(제목, URL, 조회수, 댓글수, 작성시간), 본문 수집·AI 요약 없음
  - 3분 주기
- **세이프티봇**
  - 금칙어 목록·단어 경계 매칭, 매칭 시 검토 대기
  - 사용자 문구: "등록되었습니다. 내용 검토 후 공개되거나 삭제될 수 있습니다."
  - 관리자: 일 1회 또는 10건 이상 시 배치 검토 후 공개/삭제

- 담당 A: 네이버 뉴스 API 30분 Cron, 제목·링크·출처·날짜만 저장·표시(요약문 미저장·미노출). 수집 목록 API·이슈 연결, 한도 80% 알림·주기 완화. 더쿠·네이트판 메타데이터 수집(3분)·저장·이슈 연결. 화력 분석.
- 담당 B: 토론 주제·토론 댓글·세이프티·글로벌 검색·Auth 연동. 관리자 토론 주제 승인·세이프티 검토.

**Day 9 (화력 분석)**

- 화력 분석(경량 AI 또는 자체 알고리즘), 메타데이터만 입력
- 결과는 수치이므로 관리자 승인 없이 자동 반영(02 확정)
- 담당 B: 화력 지수 표시(숫자+등급) 방식 정리(화면 문구·표시 방식)

**Day 10 (투표·커뮤니티·세이프티)**

- 투표 UI: 복수 투표·상태별 이전 투표 노출
- 커뮤니티 UI: 토론 주제 목록·상세·댓글, "해당 이슈 보기" 링크
- AI 토론 주제 생성·관리자 승인 대기
- 세이프티봇: 금칙어 매칭 → 검토 대기 → 관리자 공개/삭제 (검토 대기 안내 문구는 02 확정문구 또는 AI 요청)

**Day 11 (데이터·통합)**

- 이슈·타임라인 채우기
- 수집 → 화력 → 타임라인 반영 흐름 테스트
- 출처 링크·권한(비로그인 버튼 비활성)·관리자 전용 접근 확인

**Day 12 (배포·QA)**

- Vercel 배포, 환경 변수·도메인(whynali.com 등)
- QA: 이슈 목록·상세 → 타임라인 → 댓글·반응·투표·토론 → 관리자 수집/화력/검토
- 04·05 체크리스트(수집·노출·UGC) 확인

**Day 13 (인증·로그인)**

- 오픈ID 3종 연동
- 로그인 사용자만 감정·댓글·투표·토론 가능
- 로그인 전/후 메뉴·버튼 노출 정리, 버그 수정

**Day 14 (마감)**

- 최종 배포·URL 공유, 2주 MVP 완료
