## Part 1: 타임라인 표시 문제 해결 (완료)

### 발견된 문제

관리자 이슈 목록에서 `source_track` 필터가 제대로 적용되지 않아, 레거시 이슈(트랙 A 이전 방식으로 생성된 이슈)들이 함께 표시되고 있었습니다.

**레거시 이슈의 특징:**
- `source_track` 컬럼이 `NULL` 또는 `track_a`가 아닌 값
- 타임라인이 없거나 불완전할 수 있음
- 구 프로세스(auto-create-issue, detect-community-burst)로 생성됨

### 적용된 수정

#### 1. 관리자 API 필터 추가

파일: `app/api/admin/issues/route.ts`

```typescript

}
```

프론트엔드에서 보내는 `source_track=track_a` 파라미터를 이제 정상적으로 처리합니다.

## Part 2: 타임라인 단계 분류 개선 설계 (진행 중)

### 배경

이슈 상세의 타임라인에는 발단 / 전개 / 파생 / 진정 4단계가 있지만,
현재 track-a의 실제 코드는 `index === 0 ? '발단' : '전개'`로 하드코딩되어 있어
**파생과 진정은 DB 스키마에만 존재하고 실제로 배정된 적이 없다.**

### 현재 구조의 문제 (모순 2가지)

#### 모순 1: `auto-timeline.ts` 사문화

`lib/timeline/auto-timeline.ts`에 `assignStages()` 로직이 있지만 track-a에서 호출하지 않는다.
track-a는 자체 인라인 코드로 타임라인을 생성하며, 파생/진정은 배정하지 않는다.

→ **`auto-timeline.ts` 삭제 대상**

#### 모순 2: 이슈 생존 중 타임라인 업데이트 없음

타임라인 생성은 이슈 최초 생성 시(track-a)에만 이루어진다.
이후 새 뉴스가 수집되거나 중복 이슈가 감지되어 기존 이슈에 연결되어도 타임라인에 반영되지 않는다.

특히 중복 감지 분기에서:
- `community_data.issue_id` → 기존 이슈로 연결 ✓
- `news_data.issue_id` → 업데이트 없음 ✗
- `timeline_points` → 업데이트 없음 ✗

---

### 단계 배정 원칙

| 단계 | 판단 기준 | 방법 |
|------|-----------|------|
| 발단 | 시간순 첫 번째 기사 | 로직 (현재 유지) |
| 전개 | 원이슈의 직접적 발전 (입장 발표, 후속 조치 등) | Groq 분류 |
| 파생 | 새 인물/논란 등장, 간접 연관 별개 사건 | Groq 분류 |
| 진정 | 이슈 status = '종결'일 때 마지막 기사 | 로직 |

#### 파생 vs 새 이슈 경계

- **새 이슈로 독립** → track-a의 `checkDuplicateIssue`가 결정 (중복 X → 새 이슈 생성)
- **파생 타임라인 포인트** → 이미 기존 이슈에 `news_data.issue_id`로 귀속된 기사들 중 Groq이 분류

두 단계가 역할 분리되어 있으므로 모순 없음.

---

### 개선 방향

#### 1. track-a: 이슈 생성 시 초기 타임라인 (Groq 분류 적용)

```
첫 번째 기사 → '발단'
중간 기사들 → Groq 배치 요청으로 '전개' / '파생' 분류
마지막 기사 → 이슈 status '종결' 시 '진정', 아닌 경우 '전개'
```

**최적화:**
- 중간 기사가 2개 이하면 Groq 호출 없이 '전개' 배정 (비용 절감)
- Groq 실패 시 전부 '전개'로 fallback

#### 2. track-a: 중복 감지 분기에서 news_data 연결 추가

중복 이슈 감지 시 기존에 `community_data`만 연결했으나,
`news_data`도 AI 필터링 후 기존 이슈에 연결하도록 수정.

```
중복 감지
→ community_data: AI 필터링 후 기존 이슈 연결 (기존)
→ news_data: AI 필터링 후 기존 이슈 연결 (신규 추가)
```

필터링 없이 연결하면 관련 없는 뉴스가 섞일 위험이 있으므로 반드시 AI 필터링 적용.

#### 3. update-timeline cron 신설

활성 이슈의 새 뉴스를 주기적으로 감지하여 타임라인에 반영.

**주기:** `30 * * * *` (매시 30분)
- track-a가 정각에 실행(`0 * * * *`)되므로 30분 오프셋으로 race condition 방지

**처리 대상:**
- `점화`, `논란중` 이슈: 항상 처리
- `종결` 이슈: 최근 48시간 내 수집이 있으면 처리 (재점화 대응), 없으면 스킵

**처리 흐름:**
```
활성 이슈 조회
→ 이슈별 news_data 조회 (issue_id로 연결된 것)
→ 이미 timeline_points에 있는 source_url 제외 (중복 방지)
→ 새 뉴스가 있으면 Groq 배치 요청 → 전개/파생 분류
→ 마지막 포인트가 '진정'이면 → 새 포인트는 '전개' 또는 '파생'으로 추가
→ 이슈 status '종결'이면 마지막 포인트를 '진정'으로 재배정
```

**Race condition 방지:**
- track-a (`0 * * * *`) → 30분 뒤 update-timeline (`30 * * * *`) 순서로 실행
- news_data.issue_id가 먼저 연결된 후 update-timeline이 감지하는 구조

## 타임라인 시스템 구조

### DB 스키마

#### timeline_points 테이블

```sql
CREATE TABLE timeline_points (
    id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    issue_id UUID NOT NULL REFERENCES issues(id),
    stage TEXT NOT NULL CHECK (stage IN ('발단', '전개', '파생', '진정')),
    title TEXT NOT NULL,
    source_url TEXT NOT NULL,
    detected_at TIMESTAMPTZ NOT NULL,
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    UNIQUE (issue_id, source_url)
);
```

#### 인덱스

```sql
CREATE INDEX idx_timeline_points_issue_id ON timeline_points(issue_id);
CREATE INDEX idx_timeline_points_stage ON timeline_points(stage);
```

### 3. DB 상태 확인

```sql
-- 이슈별 타임라인 포인트 개수
SELECT 
    i.id,
    i.title,
    i.status,
    COUNT(tp.id) as timeline_count,
    STRING_AGG(tp.stage, ' → ' ORDER BY tp.detected_at) as stages
FROM issues i
LEFT JOIN timeline_points tp ON i.id = tp.issue_id
WHERE i.source_track = 'track_a'
GROUP BY i.id, i.title, i.status
ORDER BY i.created_at DESC;

-- 단계별 포인트 개수
SELECT 
    stage,
    COUNT(*) as count
FROM timeline_points
GROUP BY stage
ORDER BY 
    CASE stage
        WHEN '발단' THEN 1
        WHEN '전개' THEN 2
        WHEN '파생' THEN 3
        WHEN '진정' THEN 4
    END;
```

## 문제 해결

### 타임라인이 표시되지 않는 경우

**체크리스트:**

1. 이슈가 Track A 이슈인지 확인
```sql
SELECT source_track FROM issues WHERE id = 'issue-uuid';
```

2. timeline_points에 데이터가 있는지 확인
```sql
SELECT * FROM timeline_points WHERE issue_id = 'issue-uuid';
```

3. API 응답 확인
```bash
curl http://localhost:3000/api/timeline/issue-uuid
```

4. 브라우저 콘솔에서 로그 확인
```
[TimelineSection] Timeline data count: 0
```

### 레거시 이슈가 계속 표시되는 경우

**해결:**

1. 마이그레이션 실행
```bash
supabase db push --file supabase/migrations/009_cleanup_legacy_issues.sql
```

2. source_track 필터 확인
```typescript
// app/api/admin/issues/route.ts
if (sourceTrack) {
    query = query.eq('source_track', sourceTrack)
}
```

### Groq 분류 실패 시 (향후 구현)

**Fallback 처리:**
- 모든 중간 기사를 '전개'로 배정
