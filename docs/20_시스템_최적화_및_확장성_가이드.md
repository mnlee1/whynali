### 비개발자를 위한 설명

#### ISR (15분 캐싱) = 요리 미리 만들어두기

패스트푸드점에서 인기 메뉴는 미리 만들어서 보온함에 15분간 보관. 주문이 들어오면 바로 제공. 15분 지나면 새로 만들어서 교체.

**웹사이트 적용:**
- 페이지를 15분마다 한 번만 만듦
- 1,000명이 같은 페이지 보러 와도 한 번만 만들면 됨
- 속도: 0.5초 → 0.05초 (10배 빠름)
- 비용: 1,000번 만들기 → 1번 (1,000배 절감)

#### Edge Caching = 동네마다 편의점 두기

본사 창고가 서울에 있는데, 부산 사람이 물건 주문하면 서울에서 부산까지 배송 필요. 대신 부산 동네마다 편의점을 두고 인기 상품을 미리 배치하면, 부산 사람은 동네 편의점에서 바로 구매.

**효과:**
- 속도: 미국 왕복 0.3초 → 서울 왕복 0.05초

#### React Server Components = 짐 가볍게 싸기

여행 갈 때 가방에 옷, 신발, 화장품 다 넣으면 무거움. 현지에서 살 수 있는 건 현지에서 사고, 꼭 필요한 것만 가방에 넣으면 가벼워짐.

**효과:**
- 다운로드 크기: 200KB → 50KB (4배 감소)
- 모바일 3G: 6초 → 1.5초 (4배 빠름)

#### DB 인덱스 = 책 뒤 찾아보기

500페이지 책에서 "김연경" 단어 찾기:
- 인덱스 없음: 1페이지부터 500페이지까지 하나하나 읽기 → 10분
- 인덱스 있음: 책 뒤 "찾아보기"에서 "김연경 - 127페이지" 확인 → 10초

**효과:**
- 검색 속도: 1초 → 0.01초 (100배 빠름)

#### [P1] 서비스 오픈 전에 해야 할 것

##### P1-1. 데이터베이스 인덱스 최적화

**왜 필요한가?**

인덱스는 책의 "목차"와 같습니다.
목차가 없으면 원하는 내용을 찾으려고 책을 처음부터 끝까지 읽어야 하듯,
인덱스가 없으면 데이터베이스가 모든 데이터를 뒤져야 해서 느립니다.

**안 하면 어떻게 되나?**
- 이슈 목록, 댓글 목록 로딩이 느려집니다 (5초 이상)
- 유저가 답답함을 느끼고 이탈합니다

**어떻게 하나?**

Supabase 대시보드 > SQL Editor에서 `001_performance_optimization.sql` 파일 내용 실행:

```sql
-- 이슈 테이블 인덱스
CREATE INDEX IF NOT EXISTS idx_issues_status ON issues(status);
CREATE INDEX IF NOT EXISTS idx_issues_heat_index ON issues(heat_index DESC);
CREATE INDEX IF NOT EXISTS idx_issues_category ON issues(category);
CREATE INDEX IF NOT EXISTS idx_issues_created_at ON issues(created_at DESC);

-- 뉴스 데이터 인덱스
CREATE INDEX IF NOT EXISTS idx_news_issue_id ON news_data(issue_id);
CREATE INDEX IF NOT EXISTS idx_news_published_at ON news_data(published_at DESC);

-- 커뮤니티 데이터 인덱스
CREATE INDEX IF NOT EXISTS idx_community_issue_id ON community_data(issue_id);
CREATE INDEX IF NOT EXISTS idx_community_written_at ON community_data(written_at DESC);

-- 댓글 인덱스
CREATE INDEX IF NOT EXISTS idx_comments_issue_id ON comments(issue_id);
CREATE INDEX IF NOT EXISTS idx_comments_created_at ON comments(created_at DESC);

-- 투표 인덱스
CREATE INDEX IF NOT EXISTS idx_votes_issue_id ON votes(issue_id);
CREATE INDEX IF NOT EXISTS idx_votes_phase ON votes(phase);
```

**비용: 무료 (쿼리 실행만 필요)**

##### P1-2. API 응답 캐싱 설정

**왜 필요한가?**

같은 이슈 목록을 100명이 동시에 요청하면, 서버가 100번 계산합니다.
캐싱은 "한 번 계산한 결과를 저장해두었다가 재사용"하는 것입니다.
식당에서 미리 만들어둔 반찬을 내놓는 것과 같습니다.

**안 하면 어떻게 되나?**
- 서버가 매번 계산하느라 바빠집니다
- 응답 속도가 느려지고 Vercel 함수 실행 횟수도 늘어납니다

**어떻게 하나?**

주요 페이지 파일에 캐시 설정 추가:

**`app/page.tsx` (홈 - 이슈 목록)**
```typescript
// 파일 상단에 추가
export const revalidate = 60 // 60초마다 갱신
```

**`app/issue/[id]/page.tsx` (이슈 상세)**
```typescript
// 파일 상단에 추가
export const revalidate = 30 // 30초마다 갱신
```

개발자에게 요청:
"홈 페이지와 이슈 상세 페이지에 revalidate 캐싱을 추가해주세요."

**비용: 무료 (코드 추가만 필요)**

##### P1-3. 이미지 최적화 확인

**왜 필요한가?**

이미지는 웹사이트에서 가장 용량이 큰 자원입니다.
최적화되지 않은 이미지는 로딩 시간을 늘리고, 유저의 데이터 요금도 많이 씁니다.

**안 하면 어떻게 되나?**
- 페이지 로딩이 느립니다 (특히 모바일)
- 유저가 데이터를 많이 소모합니다
- SEO(검색 엔진 최적화) 점수가 낮아집니다

**어떻게 하나?**

현재 코드에서 `<img>` 태그 대신 Next.js의 `<Image>` 컴포넌트를 사용하고 있는지 확인:

**잘못된 예:**
```typescript
<img src="/images/logo.png" alt="로고" />
```

**올바른 예:**
```typescript
import Image from 'next/image'

<Image src="/images/logo.png" alt="로고" width={200} height={50} />
```

개발자에게 요청:
"모든 이미지가 Next.js Image 컴포넌트를 사용하는지 확인하고, 사용하지 않는 곳을 수정해주세요."

**비용: 무료 (코드 수정만 필요)**

#### [P2] 여유 있을 때 하면 좋은 것

##### P2-1. 모니터링 및 알림 설정

**왜 필요한가?**

웹사이트가 느려지거나 에러가 발생할 때, 유저가 불평하기 전에 미리 알면 빨리 대응할 수 있습니다.
건강검진처럼 서비스 상태를 주기적으로 체크하는 것입니다.

**안 하면 어떻게 되나?**
- 문제가 생겨도 모릅니다
- 유저가 불만을 쌓아두다가 이탈합니다
- 해결이 늦어집니다

**어떻게 하나?**

**1) Vercel Analytics 활성화**
1. Vercel 대시보드 > 프로젝트 선택
2. Analytics 탭 클릭
3. "Enable Analytics" 버튼 클릭
4. 페이지 로딩 속도, 방문자 수 등을 확인할 수 있습니다

**2) Supabase 대시보드 모니터링**
1. Supabase 대시보드 > 프로젝트 선택
2. Reports 탭 클릭
3. Database Health, API Usage 확인
4. Settings > Billing에서 사용량 확인

**3) 에러 추적 도구 (선택사항)**
- Sentry 같은 에러 추적 서비스 사용
- 에러가 발생하면 이메일이나 슬랙으로 알림

**비용: Vercel Analytics 무료 (Pro 플랜 포함), Sentry 무료 플랜 또는 월 $26**

### 자주 묻는 질문

**Q1. 무료 플랜으로는 절대 안 되나요?**

A. 유저 100명 이하라면 가능하지만, 1,000명 동시 접속은 불가능합니다.
무료 플랜은 취미 프로젝트나 테스트용입니다.

**Q2. Vercel Pro가 $20인데 왜 더 나올 수 있나요?**

A. Vercel Pro는 기본 $20 + 종량제 모델입니다.
- 포함 한도: 함수 호출 100만 회, 대역폭 1TB/월
- 초과하면 자동으로 추가 과금됩니다
- 왜난리 프로젝트는 함수 호출이 240만 회 예상되어 약 $1~$5 추가
- 대부분의 웹사이트는 $20~$30 사이에서 운영됩니다

**Q3. Pro 플랜으로 업그레이드하면 무제한인가요?**

A. 아닙니다. 포함 한도가 있고, 초과하면 종량제로 과금됩니다.
- 함수 호출: 100만 회/월 포함, 초과 시 100만 회당 $0.60
- 대역폭: 1TB/월 포함, 초과 시 지역별 요금
- 이미지가 많거나 동영상이 있으면 대역폭 초과 가능성 있음

**Q4. 비용을 줄일 방법은 없나요?**

A. 있습니다:
- API 캐싱으로 함수 호출 수를 줄이면 추가 비용 감소
- 이미지 최적화로 대역폭 사용량 감소
- 불필요한 API 호출 제거
하지만 Pro 플랜 자체는 1,000명 동시 접속 목표에서 필수입니다.

**Q5. 언제 업그레이드해야 하나요?**

A. 서비스 오픈 최소 1주일 전에는 완료해야 합니다.
설정 후 테스트하고, 문제가 생기면 대응할 시간이 필요합니다.

**Q6. 나중에 유저가 더 늘어나면?**

A. Vercel Enterprise 플랜(맞춤 견적)이나,
AWS, GCP 같은 클라우드로 이전을 고려해야 합니다.
하지만 10,000명까지는 Pro 플랜으로 충분합니다.

## 문제 해결

### DB 연결 에러

```
Error: remaining connection slots reserved
```

**해결:**
- Connection Pooler 활성화 확인
- `USE_SUPABASE_POOLER=true` 설정 확인

### Groq API Rate Limit

```
Error: 429 Too Many Requests
```

**해결:**
- 키 개수 확인 (최소 10개)
- 키 추가 발급

### 페이지 로딩 느림

**확인:**
- DB 인덱스 생성 여부
- ISR 캐싱 설정 여부 (revalidate: 900)
