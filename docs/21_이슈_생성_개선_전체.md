# 이슈 생성 개선 전체 히스토리

날짜: 2026-02-26

## 1. 이슈 타이틀 선정 개선

### 1.1 문제점

기존 방식: 가장 짧은 제목 선택
```typescript
function selectRepresentativeTitle(items: RawItem[]): string {
    return items
        .map((i) => stripMediaPrefix(i.title))
        .reduce((shortest, cur) => cur.length < shortest.length ? cur : shortest)
}
```

문제 사례:
```
뉴스 제목들:
1. "민희진 어도어"  ← 기존: 이게 선택됨 (가장 짧음)
2. "민희진, 256억 포기 소송 끝내자 기자회견"
3. "민희진 "어도어, '뉴진스 돌아오면 잘해주겠다'는 약속 지키길""

문제:
- 핵심 내용(256억, 소송, 기자회견) 누락
- 이슈의 본질을 알 수 없음
- 관련 뉴스 수집도 제대로 안 됨
```

### 1.2 해결 방안: 정보가 가장 많은 제목 선택

새로운 방식:
```typescript
function selectRepresentativeTitle(items: RawItem[]): string {
    const titlesWithInfo = items.map((i) => {
        const cleanTitle = stripMediaPrefix(i.title)
        const tokens = tokenize(cleanTitle)
        return {
            title: cleanTitle,
            tokenCount: tokens.length,
            length: cleanTitle.length
        }
    })
    
    // 토큰 개수 많은 순 → 길이 긴 순으로 정렬
    titlesWithInfo.sort((a, b) => {
        if (b.tokenCount !== a.tokenCount) {
            return b.tokenCount - a.tokenCount  // 토큰 많은 순
        }
        return b.length - a.length  // 길이 긴 순
    })
    
    return titlesWithInfo[0].title
}
```

선정 기준 (우선순위):
1. 키워드(토큰) 개수가 많은 제목 → 정보량이 많음
2. 키워드 개수가 같으면 길이가 긴 제목 → 더 구체적
3. 길이도 같으면 첫 번째 아이템 → 시간순

### 1.3 개선 효과

예시 1: 민희진 이슈

Before:
```
선택된 타이틀: "민희진 어도어"
→ 핵심 내용 누락
```

After:
```
선택된 타이틀: "민희진, 256억 포기 소송 끝내자 기자회견"
→ 256억, 소송, 기자회견 등 핵심 정보 포함
→ 관련 뉴스 수집 정확도 향상
```

예시 2: 갤럭시 이슈

Before:
```
선택된 타이틀: "갤럭시S26"
→ 출시 정보 누락
```

After:
```
선택된 타이틀: "삼성 갤럭시S26 공식 출시 예정"
→ 삼성, 출시, 예정 등 구체적 정보 포함
```

### 1.4 장점

1. 정보성: 핵심 내용을 포함한 타이틀
2. 수집 정확도: 구체적인 타이틀로 관련 뉴스 수집 향상
3. 사용자 이해도: 이슈 내용을 바로 파악 가능
4. 그루핑 정확도: 키워드가 많아 관련 뉴스 매칭 향상

## 2. AI 기반 중복 이슈 체크

### 2.1 문제점

기존 방식: 정확한 제목 일치만 체크
```typescript
.eq('title', representativeTitle)
```

문제 사례:
```
이슈 1: "민희진, 256억 포기 소송 종결 제안"
이슈 2: "민희진 변호사 '모든 금전 요구 포기' 발표"
→ 제목이 다름 → 중복으로 인식 안 됨 → 2개 등록
```

발생 원인:
- 제목 선정 로직 변경 (가장 짧은 → 가장 정보 많은)
- 같은 사건을 다르게 표현한 뉴스
- 언론사별 다른 헤드라인

### 2.2 해결 방안: AI 기반 중복 체크

핵심 아이디어: 사람처럼 "같은 사건인지" 판단
- 제목이 달라도 내용이 같으면 중복
- 같은 인물이지만 다른 전개면 별개
- 반대 사건은 별개

### 2.3 구현 내용

안전 장치 (빠른 제외 체크):

1. 반대어 체크
```typescript
const OPPOSITE_WORD_PAIRS = [
    ['복귀', '사퇴', '퇴사'],
    ['찬성', '반대'],
    ['승인', '반려'],
    ['기소', '무혐의'],
]

"민희진 복귀" vs "민희진 사퇴"
→ 반대어 감지 → 별개 이슈 (안전)
```

2. 숫자 차이 체크
```typescript
"민희진 1차 회견" vs "민희진 2차 회견"
→ 숫자 차이 감지 → 별개 이슈
```

3. 키워드 필터
```typescript
공통 키워드 2개 미만 → 별개 이슈 (빠른 제외)
```

AI 정밀 체크 (Groq):

프롬프트:
```
두 이슈 제목이 같은 사건인지 판단:

[기존] 민희진, 256억 포기 소송 종결 제안
[신규] 민희진 변호사 '모든 금전 요구 포기' 발표

판단 기준:
1. 완전히 같은 사건 → 중복
2. 새로운 전개 → 별개
3. 반대 사건 → 별개
4. 다른 표현 → 중복

응답: { "isDuplicate": true, "confidence": 90, "reason": "같은 내용을 다르게 표현" }
```

Confidence 임계값:
```typescript
if (confidence < 80) {
    // 확신 없으면 별개 이슈로 처리 (안전)
    return { isDuplicate: false }
}
```

### 2.4 처리 흐름

```
1단계: 정확한 제목 일치 체크
   ↓ 없으면
   
2단계: 키워드 필터 (공통 2개 미만 → 별개)
   ↓ 통과하면
   
3단계: 안전 장치 체크
   - 반대어? → 별개
   - 숫자 차이? → 별개
   ↓ 통과하면
   
4단계: AI 정밀 체크
   - Confidence 80% 이상? → 중복
   - 미만? → 별개 (안전)
   
5단계: Rate Limit/에러
   → 별개 이슈로 처리 (안전)
```

### 2.5 테스트 시나리오

시나리오 1: 같은 사건, 다른 표현
```
기존: "민희진, 256억 포기 소송 종결 제안"
신규: "민희진 변호사 '모든 금전 요구 포기' 발표"

1. 공통 키워드: 3개 (민희진, 포기, 발표) → 통과
2. 반대어: 없음 → 통과
3. 숫자 차이: 없음 → 통과
4. AI 판단: 중복 (95%) → 중복 처리
```

시나리오 2: 반대 사건
```
기존: "민희진 복귀 발표"
신규: "민희진 사퇴 발표"

1. 공통 키워드: 2개 (민희진, 발표) → 통과
2. 반대어: 복귀 vs 사퇴 → 별개 이슈 (안전 장치)
```

시나리오 3: 연속 사건
```
기존: "민희진 1차 기자회견"
신규: "민희진 2차 기자회견"

1. 공통 키워드: 3개 → 통과
2. 반대어: 없음 → 통과
3. 숫자 차이: 1 vs 2 → 별개 이슈 (안전 장치)
```

### 2.6 비용 및 성능

API 사용량:
- 이슈 등록 시 최대 20개 체크
- 안전 장치로 대부분 제외 → 실제 AI 호출 5회 이하
- 1회당 3초 대기 → 15초 추가

일일 사용량:
- 시간당 이슈: 5~10개
- 이슈당 AI 호출: 평균 3회
- 일당 총 호출: 360~720회 (RPD 14,400 안에 충분)

커뮤니티 매칭과 합산:
- 커뮤니티: 최대 10회/이슈
- 중복 체크: 평균 3회/이슈
- 합계: 13회/이슈 × 120 이슈/일 = 1,560회/일

### 2.7 장점

1. 정확한 중복 감지: 같은 사건을 다르게 표현해도 감지
2. 안전 장치: 반대 사건, 연속 사건 보호
3. 보수적 처리: 낮은 신뢰도 거부 (80% 미만)
4. 무료: Groq API 무료
