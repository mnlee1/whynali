## 2. 중복 이슈 자동병합 가이드

### 2.1 문제 정의

**현상:**
```
이슈 1: "민희진 하이브와 전면전"
  - 뉴스 5건
  - 화력 45점
  
이슈 2: "어도어 대표직 사수"
  - 뉴스 3건
  - 화력 32점
  
→ 실제로는 같은 "민희진-하이브 논란" 이슈
```

**원인:**
1. 제목 표현 차이 - 공통 키워드 부족 (민희진/하이브 vs 어도어/대표직)
2. 시간대별 수집 - 24시간 창에서 뉴스가 여러 번 수집되면서 그루핑 시점 차이
3. 키워드 임계값 - 공통 키워드 1개 이상 조건으로는 부족

### 2.2 해결 방안

**아키텍처:**
```
[뉴스 수집]
    ↓
[키워드 그루핑] (공통 키워드 1개 이상)
    ↓
[AI 재검증] ← NEW
    ↓ (같은 카테고리 + 24시간 이내)
    ↓
[Groq AI 비교] (신뢰도 80%+)
    ↓
[자동 병합]
    ↓
[이슈 등록]
```

### 2.3 AI 중복 체크 로직

**4단계 검증:**
1. 정확한 제목 일치 (빠른 체크)
2. 키워드 필터링 (공통 키워드 2개 이상)
3. 안전 장치 (반대어, 숫자 차이 감지)
4. Groq AI 정밀 비교 (신뢰도 80% 이상)

**안전 장치:**

반대어 감지:
- "복귀" vs "사퇴" → 별개
- "기소" vs "무혐의" → 별개

숫자 차이 감지:
- "1차 회견" vs "2차 회견" → 별개
- "재판 3일차" vs "4일차" → 별개

### 2.4 환경변수

```bash
# AI 중복 체크 활성화
ENABLE_AI_DUPLICATE_CHECK=true

# Groq API
GROQ_API_KEY=your_groq_api_key

# 신뢰도 임계값
DUPLICATE_CHECK_AI_CONFIDENCE=80
```

## 3. 중복 이슈 원인 분석 및 해결 방안

### 3.1 중복 발생 원인

**Case 1: 키워드 부족**
```
이슈 A: "민희진 vs 하이브"
이슈 B: "어도어 대표 vs 방시혁"
→ 공통 키워드: 0개
```

**Case 2: 시간차 수집**
```
12:00 - 뉴스 3건 수집 → 이슈 A 생성
13:00 - 뉴스 5건 수집 → 이슈 B 생성
→ 같은 주제지만 다른 시점에 생성
```

**Case 3: 표현 차이**
```
"옥택연 결혼" vs "옥택연 웨딩"
"민희진 논란" vs "민희진 사태"
→ 같은 의미, 다른 표현
```

### 3.2 해결 방안

**기존 이슈 병합:**
```
1. 관리자 화면에서 중복 감지
2. AI로 자동 비교 (신뢰도 80% 이상)
3. 병합 버튼 클릭
4. 자동 병합 처리:
   - 뉴스/커뮤니티 데이터 이동
   - 화력 재계산
   - 구 이슈 삭제
```

**자동 병합 로직:**
```typescript
// 같은 카테고리 + 24시간 이내 이슈 검색
const candidates = await findSimilarIssues({
    category: issue.category,
    within: '24 hours'
})

// AI로 중복 체크
for (const candidate of candidates) {
    const result = await checkDuplicate(issue, candidate)
    if (result.confidence >= 80) {
        await mergeIssues(issue, candidate)
    }
}
```

## 4. DB 마이그레이션 실행 가이드

### 4.1 병합 관련 테이블 추가

```sql
-- 이슈 병합 히스토리 테이블
CREATE TABLE IF NOT EXISTS issue_merge_history (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    target_issue_id UUID NOT NULL REFERENCES issues(id) ON DELETE CASCADE,
    source_issue_id UUID NOT NULL,
    source_title TEXT NOT NULL,
    merged_at TIMESTAMPTZ DEFAULT NOW(),
    merged_by UUID REFERENCES auth.users(id),
    confidence INTEGER,
    reason TEXT
);

-- 인덱스
CREATE INDEX IF NOT EXISTS idx_merge_history_target ON issue_merge_history(target_issue_id);
CREATE INDEX IF NOT EXISTS idx_merge_history_source ON issue_merge_history(source_issue_id);
```

### 4.2 급증 감지 관련 컬럼 (이미 적용됨)

```sql
ALTER TABLE issues 
ADD COLUMN IF NOT EXISTS is_urgent BOOLEAN DEFAULT FALSE,
ADD COLUMN IF NOT EXISTS burst_level INTEGER DEFAULT 0,
ADD COLUMN IF NOT EXISTS source_track TEXT;

CREATE INDEX IF NOT EXISTS idx_issues_is_urgent ON issues(is_urgent) WHERE is_urgent = true;
```

## 6. 모니터링

### 6.1 병합 히스토리 조회

```sql
-- 오늘 병합된 이슈
SELECT 
    t.title as target_title,
    h.source_title,
    h.confidence,
    h.merged_at
FROM issue_merge_history h
JOIN issues t ON h.target_issue_id = t.id
WHERE h.merged_at >= CURRENT_DATE
ORDER BY h.merged_at DESC;

-- 카테고리별 병합 통계
SELECT 
    i.category,
    COUNT(*) as merge_count
FROM issue_merge_history h
JOIN issues i ON h.target_issue_id = i.id
WHERE h.merged_at >= CURRENT_DATE - INTERVAL '7 days'
GROUP BY i.category;
```
