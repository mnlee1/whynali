# 데이터베이스 백업 및 복구 가이드

왜난리 서비스의 Supabase DB를 안전하게 백업하고 복구하는 방법을 설명합니다.

## 개요

세 가지 백업 전략을 함께 사용합니다.

1. **자동 백업 시스템** - 매일 오전 9시 GitHub Actions 실행 (public.users, 최근 7일 보관)
2. **Auth 사용자 백업** - 주 1회 실행 (auth.users, 로그인 정보 백업)
3. **Supabase 복구** - 프로젝트 삭제 시 대시보드에서 복구 (7일 이내)

**백업 대상**: 실서버(whynali-main) - `mdxshmfmcdcotteevwgi.supabase.co`

**환경 분리**:
- 개발 서버(whynali-dev): `.env.local` (로컬 개발용)
- 실서버(whynali-main): `.env.production.local` (백업용)

**주의**: 저장소가 Public이므로 기본 백업 파일은 공개됩니다. Auth 백업은 로컬에만 보관하세요.

**두레이 알림**: 매주 월요일 오전 10시에 Auth 백업 알림이 전송됩니다.

## 자동 백업 시스템 (Public 저장소용)

### 작동 방식

매일 오전 9시 (한국시간) GitHub Actions가 자동 실행됩니다.

1. Supabase에서 핵심 테이블 데이터 가져오기
2. JSON 파일로 저장 (이메일 제외)
3. GitHub Public 저장소에 자동 커밋
4. 7일 이전 백업 자동 삭제

### 백업되는 테이블

- issues (이슈)
- users (사용자 - 이메일 제외)
- comments (댓글)
- reactions (반응)
- votes (투표)
- discussion_topics (토론)
- news_data (뉴스 데이터)
- timeline_points (타임라인)

### 제한사항

⚠️ **auth.users (로그인 정보)는 백업되지 않습니다**
- Supabase Auth가 관리하는 테이블
- 이메일, 비밀번호, OAuth 연동 정보 포함
- 프로젝트 삭제 시 함께 삭제됨

→ **해결책**: Auth 백업 스크립트 별도 실행 (아래 참고)

### 백업 파일 위치

```
backups/
  2026-04-09/
    issues.json
    users.json
    comments.json
    _meta.json
```

### 비용

완전 무료입니다.

- GitHub Actions: Public 저장소 무제한 무료
- 사용량 제한 없음

## 백업에서 복원

### 복원 시나리오 이해하기

#### 시나리오 1: 단순 데이터 손실 (✅ 완전 복구 가능)
- 상황: 테이블 삭제, 잘못된 업데이트 등
- 보존: auth.users (이메일, 비밀번호) 그대로 유지
- 복원: public.users만 복원하면 OK
- 영향: contact_email만 null (사용자가 마이페이지에서 다시 입력)

#### 시나리오 2: Supabase 프로젝트 완전 삭제 (✅ Auth 백업 있으면 복구 가능!)
- 상황: 프로젝트 삭제로 auth.users + public.users 모두 날아감
- **Auth 백업 있음**: 사용자 계정 복원 가능 (비밀번호 재설정 필요)
- **Auth 백업 없음**: 사용자 계정 복구 불가능 (모두 새로 가입)

### 일반 데이터 복원

```bash
# public.users, issues, comments 등 복원
node scripts/restore-db.mjs 2026-04-09
```

### Auth 사용자 복원

Supabase 프로젝트가 완전 삭제된 경우:

```bash
# 1. 새 Supabase 프로젝트 생성
# 2. 스키마 복원 (supabase/dev_setup.sql)
# 3. Auth 사용자 복원
node scripts/restore-auth-users.mjs 2026-04-09

# 4. 일반 데이터 복원
node scripts/restore-db.mjs 2026-04-09
```

**중요 안내**:
- ✅ 사용자 이메일, OAuth 연동 복원됨
- ❌ 비밀번호는 복원 안 됨 (해시를 옮길 수 없음)
- 📧 사용자에게 "비밀번호 재설정" 메일 발송 필요
- 🔑 OAuth 사용자는 다시 로그인하면 자동 연결

### 특정 테이블만 복원

```bash
node scripts/restore-db.mjs 2026-04-09 comments
```

### 복원 과정

1. 백업 파일 확인
2. 확인 메시지에 `yes` 입력
3. 자동으로 데이터 복원 (100개씩 배치 처리)

## Supabase 프로젝트 복구

프로젝트가 실수로 삭제된 경우 복구 방법입니다.

### 복구 방법 선택

| 상황 | 복구 방법 | 성공 가능성 |
|------|----------|------------|
| 7일 이내 삭제 | 대시보드 복구 | 높음 |
| 7일 경과 (Free) | Support 문의 | 낮음 |
| 7일 경과 (Pro+) | Support 문의 | 중간 |
| 완전 삭제 | 새 프로젝트 생성 | 해당없음 |

### 대시보드 복구 (7일 이내)

**1단계: 삭제된 프로젝트 찾기**

1. https://supabase.com/dashboard 접속
2. 프로젝트 목록에서 "삭제된 프로젝트" 탭 확인
3. Settings > General에서 "Restore" 옵션 확인
4. "Restore this project" 버튼 클릭
5. 복구 완료 (몇 분 소요)

**2단계: 환경변수 원복**

.env.local 파일에서 기존 프로젝트 정보 활성화:

```
NEXT_PUBLIC_SUPABASE_URL=https://your-project-ref.supabase.co
NEXT_PUBLIC_SUPABASE_ANON_KEY=your-anon-key-here
SUPABASE_SERVICE_ROLE_KEY=your-service-role-key-here
```

**3단계: 연결 테스트**

```bash
npm run dev
```

### Support 문의 (7일 경과)

**문의 방법**

- https://supabase.com/dashboard/support
- Email: help@supabase.io

**플랜별 복구 가능성**

- Free 플랜: 7일 보관, 이후 완전 삭제
- Pro/Team 플랜: Point-in-Time Recovery (PITR) 지원, 복구 가능성 높음

**1단계: GitHub 저장소 접속**

첫 번째:
- Name: `NEXT_PUBLIC_SUPABASE_URL`
- Value: .env.local 파일에서 복사 (https://xxx.supabase.co)

두 번째:
- Name: `SUPABASE_SERVICE_ROLE_KEY`
- Value: .env.local 파일에서 복사 (eyJhb... 긴 문자열)

**4단계: 테스트**

Actions 탭에서 "DB Backup" 수동 실행하여 확인

## 데이터 보안

**중요**: 저장소가 Public이므로 백업 파일이 공개됩니다!

### 백업에서 제외되는 민감 정보 (Public 저장소)
- 사용자 이메일 (contact_email) - 백업 시 자동 제외
- OAuth 제공자 ID (provider_id) - 백업 시 자동 제외

**이유**: GitHub Public 저장소에 개인정보 노출 방지

### 백업되는 안전한 데이터
이슈, 댓글, 투표 (공개 정보), 사용자 닉네임, UUID

### 주의사항
- OAuth 이메일은 auth.users에 저장 (자동 백업 불가)
- .env.local은 절대 git 커밋 금지
- backups-auth/ 폴더 절대 git 커밋 금지

### Public 저장소를 사용하는 이유

저장소를 Public으로 유지하는 것이 권장됩니다.

**이유**:
- GitHub Actions: Public 저장소 무제한 무료
- Private 전환 시: 월 2,000분 제한 (현재 사용량 월 45,000분으로 초과)
- 초과 비용: 월 약 $344

**보안**:
- 민감한 Auth 데이터는 로컬에만 보관
- 일반 백업 파일은 공개되어도 안전 (이메일 제외됨)

## 로컬 백업 파일 관리

### 자동 정리

오래된 백업 파일을 정기적으로 정리하세요:

```bash
npm run backup:cleanup
```

**보관 기간**:
- 일반 백업 (backups/): 30일
- Auth 백업 (backups-auth/): 90일 (더 중요)

### 수동 관리

**용량 확인**:
```bash
du -sh backups/ backups-auth/
```

**오래된 백업 삭제**:
```bash
rm -rf backups/2026-03-*
rm -rf backups-auth/2026-02-*
```

### 권장 관리 전략

1. **로컬 보관**
   - 일반 백업: 30일 (자동 정리)
   - Auth 백업: 90일 (자동 정리)
   - 월 1회 정리 스크립트 실행

2. **외부 보관** (중요!)
   - Auth 백업: Google Drive에 매주 업로드
   - 영구 보관용
   - 압축하여 저장 (용량 절약)

3. **정리 자동화**
   - 매월 1일 정리 스크립트 실행
   - 달력 알림 설정 또는
   - Cron 작업 추가 (선택)
