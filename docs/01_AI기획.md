# AI 기획 문서 (참조용)

본 문서는 서비스 기획을 AI가 파싱·참조하기 쉽게 구조화한 단일 기획 스펙이다. 기능·옵션·용어와 함께 기술 스택·DB·서버 정책을 포함하며, 로드맵 작성의 기준이 된다.

**서비스 구현 시 규칙 (법적 이슈)**
구현 단계에서 법적 이슈가 있을 수 있는 항목은 (1) 해당 기능을 제외하거나, (2) 해당 포인트를 사용자에게 질문하여 확인 후 진행한다. 법적 리스크가 의심되는 부분은 임의로 포함하지 않는다.

---

## 1. 서비스 한 줄 요약

한국에서 터진 이슈(사건/논란)를 한곳에 모아, 시간 순서대로 발단·전개·파생·진정을 보여주고, 사용자가 반응(감정 표현, 댓글, 투표)을 남길 수 있는 웹 서비스.

---

## 2. 배경·목적·방향

### 2.1 배경

- 한국 사용자는 연예·스포츠·정치·사회·기술 등 전 분야 이슈에 관심이 높고, 발생 직후 빠르게 정보를 탐색하며 댓글·공감 등 반응을 남기는 문화가 강함.
- 이슈 정보는 뉴스/커뮤니티/SNS에 분산되어 있어 "지금 무슨 일이 벌어졌고 왜 커졌는지"를 한 번에 파악하기 어려움.
- 이슈의 진행 과정(발단→전개→파생→진정)을 시계열로 정리해 주는 서비스에 대한 잠재 수요가 존재함.

### 2.2 목적

- 국내에서 발생하는 다양한 분야의 이슈를 한 곳에 모아, 사용자가 빠르게 맥락과 핵심을 이해하도록 돕는 것.
- 이슈의 흐름과 여론 변화를 시간축으로 정리해 "현재 상황"과 "왜 이렇게 되었는지"를 동시에 제공.
- 정보 탐색 비용을 줄이고, 댓글·리액션 등 손쉬운 반응/피드백을 통해 재방문을 유도함.

### 2.3 방향

- 단순 기사 모음이 아니라, 이벤트 단위로 묶어 타임라인과 이슈 포인트를 제공하는 **이슈 내비게이션**형 서비스로 설계.
- 해당 서비스만으로 한국의 현재·과거 이슈(연예, 스포츠, 정치, 사회 등)를 모두 확인·파악 가능해야 함.
- 하나의 이슈에 대해서도 여론 변화, 당사자 입장에 따른 상황 변화를 타임라인별로 쉽게 파악 가능해야 함.
- 이슈 주목도 하락 시 종료(Closed) 처리하고, 종료된 이슈는 과거 히스토리로 검색·열람 가능하게 운영.
- 접근성: 웹/앱 동시 제공을 염두에 두고, 로그인은 댓글 등 필요 기능에만 선택적으로 적용.
- **모바일 퍼스트**: 이슈·타임라인 소비는 모바일 비중이 높을 것으로 가정. 모바일 화면에서 한 눈에 상황을 파악할 수 있게 설계한다.

### 2.4 디바이스·화면 정책

- **모바일 퍼스트**: 기획·디자인·퍼블리싱 시 모바일 화면을 기준으로 우선 설계. 이슈 목록, 3줄 요약, 타임라인 카드, 반응 버튼 등이 모바일 한 화면에서 한 눈에 들어오도록 구성.
- **반응형 구현**: 별도 모바일 전용 사이트가 아니라 **반응형 웹**으로 구현. 한 코드베이스로 모바일·태블릿·데스크톱 대응, 유지보수와 일관된 UX 유지에 유리. Next.js 기반으로 미디어 쿼리·반응형 레이아웃 적용.
- **우선 순위**: 작은 화면(모바일)에서 정보 밀도·가독성·조작 가능 여부를 먼저 확보한 뒤, 넓은 화면으로 확장하는 순서로 진행.

## 3. 용어 정리

| 용어 | 의미 |
|------|------|
| 이슈 | 하나의 사건/논란 단위. 예: "OO 논란", "OO 사건" |
| 타임라인 | 그 이슈가 시간 순으로 어떻게 흘러갔는지 나열한 것 |
| 발단→전개→파생→진정 | 사건의 시작 → 퍼짐 → 파생 이슈 발생 → 진정·마무리 단계 |
| LifeCycle | 사건의 생명 주기. 발단→전개→파생→진정 흐름을 시각화한 개념 |
| MVP | Minimum Viable Product. 처음 출시할 때 꼭 있어야 할 기능만 넣은 버전 |
| 오픈ID | 구글, 네이버, 카카오 등으로 "다른 사이트 계정으로 로그인"하는 방식 |

## 4. 주요 기능 (MVP vs 확장)

### 4.1 MVP(2주)에 포함

- **모니터링·수집**:
  - 뉴스: 네이버 뉴스 API 1시간(또는 30분) 단위 수집. API 응답 중 제목, 링크, 출처, 날짜만 수집·표시 (요약문은 미수집. 04 4.3.1).
  - 커뮤니티: 더쿠, 네이트판 2곳만 메타데이터 수집(제목, URL, 조회수, 댓글수, 작성시간). 본문 수집 및 AI 요약 없음.
- **화력 분석**: 수집된 메타데이터(조회수, 댓글수, 반응 속도 등)를 AI로 분석하여 이슈 화력 지수 산출. 본문 요약·재가공 없음.
- **타임라인 제공**: 이슈별 관련 출처 링크를 타임라인 형태로 제공. 네이버 API 데이터는 제목·링크·출처·날짜만 표시(요약문 미수집). 추가 AI 가공 없음.
- **출처·신뢰성**: 이슈 관련 기사/콘텐츠 출처 링크를 모아 표시. 원문 링크 제공으로 원 사이트 트래픽 기여.
- **반응·참여**: 감정 표현(좋아요/싫어요 등), 댓글(좋아요·싫어요, 상단 베스트 score 상위·아래 리스트 최신순/좋아요순/싫어요순), 이슈당 복수 투표 가능(시점/상황별 여론 변화 반영, 선택지 2~4개, 실시간 결과).
- **커뮤니티**(메뉴명 "커뮤니티"): 관리자 승인 이슈 기반으로 AI가 토론 주제 후보 생성 → 관리자 승인 후 등록. 등록된 주제에 대해 유저가 철학적 관점에서 댓글로 의견 참여. 투표만 있는 구조가 아니라 글로 참여 가능.
- **세이프티봇**: 댓글·토론글 저장 시 금칙어 필터 → 자동 비공개 또는 검토 대기. 관리자 검토 후 공개/삭제.
- **관리자**: 이슈/타임라인 CRUD, 수집 기사·커뮤니티 메타데이터 확인·이슈 연결, 화력 분석 결과 승인. 과도한 리소스 투입 없이 사람이 개입하는 수준.

### 4.2 확장(2주 이후)

- **개인화**: 유저의 조회·참여 이슈 분석 → 메인 화면에 추천 이슈 우선 노출.
- **알림**: 유저가 관심 있는 이슈에 특정 이벤트 발생 시 알람 발송.
- **숏폼 자동 생성·배포**: 인스타 릴스, 유튜브 쇼츠, 틱톡 등 최소 3곳 자동 등록, 본 서비스로 유입 유도.
- **메타(페이스북) 로그인**: 기획원본에서 고민 수준이므로 필요 시 추가.

## 5. 이슈 수집 방법

### 5.1 실시간 뉴스

- 네이버 뉴스 API 활용.
- 수집 주기: 1시간 또는 30분.
- API 응답 중 제목, 링크, 출처, 날짜만 수집·표시 (요약문 미수집. 04 4.3.1). 추가 AI 가공 없음.
- 네이버 뉴스 카테고리와 이슈 카테고리 매칭 가능 여부 검토; 부적절하면 이슈별 카테고라이징 필요.
- **AI 3줄 요약과 법적 이슈**: 기사 본문이나 네이버 API 요약문을 AI에 넣어 "구단 측은 공식 입장을 내놓지 않았지만…" 같은 **서술형 3줄 요약**을 만들면, 저작권(복제권·2차적 저작물) 및 네이버 API 약관("검색 결과 조작 금지") 위반 리스크가 있음. 판례상 뉴스 요약문도 저작물로 보는 경우가 있어 출처 링크만으로는 면책되지 않음. 따라서 본문·요약문 기반 AI 요약은 하지 않고, **메타데이터만**(건수, 조회수, 반응 정도 등)을 쓴 사실 나열형("OO 논란. 관련 뉴스 5건, 반응 급상승.")만 사용하거나 API 데이터를 그대로 표시함. 상세는 04_서비스노출_법적검토.md 섹션 4 참조.

### 5.2 실시간 커뮤니티

- 대상: 더쿠, 네이트판 2곳만 (법적 리스크 최소화).
- 수집 주기: 1~5분(글 리젠 속도 대응).
- 수집 데이터: 메타데이터만(제목, URL, 조회수, 댓글수, 작성시간). 본문 수집 및 AI 요약 없음.
- 저작권 리스크 최소화, 원문 링크 제공으로 원 사이트 트래픽 기여.

## 6. 로그인/회원가입

- 모든 로그인은 오픈ID로 처리.
- MVP 지원: 구글, 네이버, 카카오. 메타는 확장 시 검토.
- 댓글·감정 표현·투표·커뮤니티 참여는 로그인 후 사용. 이슈 조회·타임라인 열람은 비로그인 가능.

---

## 7. 유저 참여 (로그인 필요)

### 7.1 감정 표현

- 좋아요, 싫어요, 화나요, 팝콘각/흥미진진, 애도/응원, 사이다/통쾌. 전 옵션 MVP 구현.

### 7.2 댓글

- 댓글에도 좋아요/싫어요 제공.
- **상단 베스트 댓글**: 좋아요−싫어요(score) 기준 상위를 상단에 배치. **아래 댓글 리스트**: 정렬 옵션 최신순, 좋아요순, 싫어요순 중 선택.

### 7.3 투표

- 논란 이슈에 대해 실시간 투표 제공.
- 이슈당 복수 투표 가능. 시점/상황에 따라 여론이 변할 수 있으므로 한 이슈에 여러 투표를 두고 시점별·상황별 추적.
- 선택지 2~4개, 결과 즉시 그래프/비율 표시.
- **이슈 상태별 이전 투표 노출**: 점화/논란중/종결 등 이슈의 현재 상태와 관계없이, 과거 시기(이전 상태)에 진행된 투표의 문항·결과를 조회할 수 있는 화면 구성. 현재 진행 중인 투표는 상단 강조, 이전 투표는 상태별 블록 또는 시간순 "이전 투표 보기" 영역으로 제공. (상세는 02_AI기획_판단포인트.md "복수 투표·이슈 상태별 화면 구성" 참조.)

### 7.4 커뮤니티 (메뉴명 "커뮤니티")

- **역할**: 관리자 승인된 이슈에서 파생된 토론 주제를 **AI가 생성**하고 **관리자 승인** 후 등록. 등록된 주제에 대해 **철학적 관점**에서 유저가 댓글로 의견을 나누는 공간.
- **흐름**: 승인 이슈 → AI 토론 주제 후보 생성(메타데이터만 사용, 본문 미사용) → 관리자 승인 → 커뮤니티에 등록 → 유저 댓글로 참여.
- **목록·플로우(MVP 둘 다 구현)**: (A) 이슈 상세에서 "이 이슈의 커뮤니티" → 해당 이슈 토론 주제 목록. (B) 커뮤니티 메뉴 → 전체 토론 주제 목록 → 토론 주제 상세. 토론 주제 상세에서 "해당 이슈 보기" 링크로 이슈 상세 페이지 이동 가능.
- **철학적 관점 유도**: 안내 문구·주제 톤·placeholder 외에 **직관적 UI** 적용 권장. 영역 시각 구분(이슈 댓글과 다르게), 아이콘·라벨, (선택) 관점 선택 칩·질문 스타터 칩, 주제 인용 스타일. **토론 흐름 시각화**: 시간선 레이아웃·답글 연결선 트리·관점별 요약 등으로 "논의가 어떻게 흘러가는지" 표현. 상세는 02 "철학적 관점 유도 — 직관적 UI" 및 "영역 시각 구분"·"토론 흐름 시각화" 참조.
- 유저 댓글에 세이프티봇 검토 적용.

## 8. 타임라인 시각화 (사건의 기승전결)

- 사용자 니즈: "그래서 누가 잘못했는데?", "이게 어떻게 시작된 거야?" 등.
- 최신순 나열이 아니라 사건의 **LifeCycle**을 시각화.
- 이슈 클릭 시: 위→아래 또는 좌→우 시간 순으로 **발단 > 전개 > 절정(현재) > 결말** 구조의 타임라인 표시.
- 각 타임라인 포인트: 출처 링크 표시(네이버 뉴스, 더쿠, 네이트판 등). 네이버 API는 제목·링크·출처·날짜만(요약문 미수집). 커뮤니티는 제목+메타데이터+링크만.
- 이슈 상세 페이지 최상단에 화력 분석 결과 먼저 배치, 관련 출처는 하단 배치(정보 경량화).
- **상태 표시**: 점화(Ignited), 논란 중(Debating), 종결(Closed). 표시는 텍스트 우선.

---

## 9. 개인화·알림 (확장)

- 유저의 조회(클릭)·참여 이슈를 분석해 메인 화면에 **추천 이슈** 우선 노출(별도 탭/영역 구성 여부 검토).
- **알림**: 유저가 관심 있는 이슈에 특정 이벤트 발생 시 알람 발송.

---

## 10. 숏폼 자동 생성·배포 (확장)

- 인스타 릴스, 유튜브 쇼츠, 틱톡 최소 3곳에 자동 등록.
- 3개 플랫폼에 본 서비스 전용 채널 생성. AI 툴로 하루 수십 개 숏폼 자동 생성 예정.
- 생성 트리거: 이슈 생성 시, 이슈의 특정 전환 시점 발생 시.
- 숏폼을 통해 본 서비스로 유입 유도.

## 11. 관리자 기능

- 관리자 기능에 과도한 리소스 투입은 불필요하다고 가정. 이슈가 될 만한 포인트에 **사람이 개입**하는 수준이 적절.
- 실시간 수집 이슈의 등록 여부 결정은 사람이 수행.
- AI 가공 콘텐츠의 릴리스 직전 목록을 관리자에게 알람으로 전달하고, **승인 버튼 클릭** 후 릴리스하는 플로우.
- **운영 정책**: 통신비밀보호법·정보통신망법상 불법정보·권리침해 대응(신고 절차, 삭제 기준 등)은 서비스 오픈 전 운영 단계에서 수립.

## 12. 옵션 정의 (구현 시 기준)

구현 시 선택지가 갈리는 항목은 아래 값을 기준으로 한다.

| 항목 | 값 |
|------|-----|

| 감정 표현 | 좋아요, 싫어요, 화나요, 팝콘각/흥미진진, 애도/응원, 사이다/통쾌. 전 옵션 구현. |
| 댓글 정렬 | 상단 베스트(score 상위). 아래 리스트: 최신순/좋아요순/싫어요순 중 선택. |
| 투표 | 이슈당 복수 투표 가능(시점/상황별 여론 변화), 선택지 2~4개, 실시간 그래프/비율. |
| 타임라인 배치 | 위→아래 또는 좌→우 시간순. 발단→전개→파생→진정 구조. 카드 형태 + AI 3줄 요약. |
| 수집 소스 | 뉴스: 네이버 뉴스 API, 1시간 또는 30분. 커뮤니티: 더쿠·네이트판 2곳, 1~5분. (확장 시 추가 소스는 법적 검토 후) |
| AI | 기사 → 3줄 요약. 여러 기사 → 타임라인 포인트 초안. API: Perplexity 등 사용량 기반 검토. |
| 세이프티봇 | 금칙어/키워드 목록 필터. 매칭 시: 자동 비공개 또는 검토 대기. 관리자: 검토 후 공개 또는 삭제. |
| 디바이스·화면 | 모바일 퍼스트. 반응형 웹으로 구현(한 코드베이스). 모바일에서 한 눈에 파악 가능하도록 우선 설계. |

## 13. 기술 스택·DB·서버 정책

구현 시 사용할 기술과 환경을 정책으로 고정한다. 로드맵은 이 정책을 전제로 일정·산출물을 배치한다.

### 13.1 기술 스택

| 구분 | 기술 | 비고 |
|------|------|------|
| 프론트엔드 | Next.js 14+ (App Router) | React 기반, SSR/SSG(SEO), Vercel 최적화, AI 코드 예제 풍부 |
| 백엔드/DB | Supabase (PostgreSQL) | DB + REST API 자동 생성, 실시간 기능, 복잡 쿼리 지원 |
| 인증 | Supabase Auth | 구글, 네이버, 카카오 오픈ID 통합 |
| 배포 | Vercel | Next.js 제작사, GitHub 연동 자동 배포 |
| 버전 관리 | GitHub | Private 저장소, 2명 협업, PR/이슈 관리 |
| 스케줄러 | Vercel Cron Jobs | 1시간(또는 30분) 단위 뉴스 수집, Next.js API Route 기반 |
| 커뮤니티 수집 | Supabase Edge Functions + Cheerio | 1~5분 주기 수집, Node.js 기반 스크래핑 |
| AI 요약·타임라인 | Perplexity API | 기사 3줄 요약, 타임라인 포인트 초안 생성. 사용량 기반 유료 |
| 외부 API | 네이버 뉴스 API, 네이버/카카오/구글 로그인 | 뉴스 수집·소셜 로그인 |
| 도메인 | whynali.com, whynaly.com | AWS 등록분, 배포 시 Vercel 연결 또는 이관 |

**비용 원칙**: 서버·배포·DB는 무료 플랜 범위에서 운영. 화력 분석은 자체 알고리즘 사용 시 무료, 경량 AI 사용 시 월 수천원 이하.

### 13.2 DB 정책

- **플랫폼**: Supabase(PostgreSQL). 스키마 설계 후 테이블 생성, API는 Supabase 자동 생성 또는 서버리스 함수로 보완.
- **저장 대상(개념)**: 이슈, 타임라인 포인트, 댓글, 반응(감정 표현), 유저(인증 연동), 투표·투표 선택지, 커뮤니티 토론 주제(이슈 연결·AI 생성·승인 상태)·토론 댓글, 수집 기사(네이버 API 데이터), 수집 커뮤니티 메타데이터(더쿠, 네이트판), 금칙어/설정 등. 본문 저장 없음. 상세 스키마(테이블·필드)는 기획 정리 후 "DB에 넣을 데이터" 목록을 바탕으로 로드맵 단계에서 설계.
- **진행 순서**: 기획 정리 → DB(스키마) 설계 → 기능 구현(API) → 화면 붙이기. DB를 먼저 정한 뒤 기능·화면 순으로 진행.

### 13.3 서버·배포 정책

- **호스팅**: Vercel. Next.js 프로젝트 배포, GitHub 연동 시 push 기준 자동 배포.
- **무료 범위**: Vercel 월 100GB 대역폭, Supabase 500MB·월 5만 API 요청, Edge Functions 50만 호출/월 등 제한 내 사용.
- **도메인**: whynali.com, whynaly.com 중 선택해 배포 환경에 연결. DNS는 기등록 상태이므로 이관 또는 CNAME 등 설정만 수행.

### 13.4 환경 변수 (필수)

로컬 및 배포(Vercel) 환경에 등록할 항목.

- SUPABASE_URL, SUPABASE_ANON_KEY
- NAVER_CLIENT_ID, NAVER_CLIENT_SECRET (뉴스 API + 네이버 로그인)
- KAKAO_CLIENT_ID, KAKAO_CLIENT_SECRET
- GOOGLE_CLIENT_ID, GOOGLE_CLIENT_SECRET

사전 준비: 네이버 검색 API(뉴스) 애플리케이션 신청, 네이버·카카오·구글 로그인 개발자 등록·키 발급. 승인·설정에 며칠 걸릴 수 있으므로 프로젝트 시작 전 처리 권장.

**화력 분석용 AI (선택)**: 경량 AI 사용 시 API 키 추가 필요 (월 수천원 이하). 자체 알고리즘 사용 시 불필요.

## 14. MVP 범위 요약

- **포함**: 이슈 목록/상세, 화력 분석 + 타임라인(상태 포함), 오픈ID 3종, 감정 표현·댓글(베스트 상단+리스트 최신순/좋아요순/싫어요순), 투표(이슈당 복수 가능, 시점/상황별), 뉴스 수집(네이버 API 데이터 그대로)·커뮤니티 메타데이터 수집(더쿠, 네이트판), 화력 분석·관리자 승인, 커뮤니티(메뉴·AI 토론 주제 생성·관리자 승인·철학적 토론), 세이프티봇(금칙어 필터·검토 대기), 관리자 CRUD·수집 확인·이슈 연결·화력 분석 승인, 웹 배포.
