# 시스템 규모 검토 (유저 10,000명, 동시접속자 1,000명)

> 작성일: 2026-03-13
> 대상 시나리오: 총 유저 10,000명, 피크타임 동시접속자 1,000명

## 1. 검토 개요

왜난리 프로젝트의 시스템 아키텍처가 유저 수 10,000명, 동시접속자 1,000명 규모를 안정적으로 처리할 수 있는지 검토합니다.

### 1.1 현재 기술 스택

- 프론트엔드: Next.js 15.1.6 (App Router), React 19
- 백엔드/DB: Supabase (PostgreSQL)
- 배포: Vercel
- AI: Groq API (무료, 다중 키), Perplexity (선택)
- 크론잡: GitHub Actions / Vercel Cron

### 1.2 트래픽 시나리오

동시접속자 1,000명 기준:
- 평균 세션 시간: 10분
- 시간당 세션: 약 6,000 세션
- 페이지뷰: 시간당 60,000 PV (사용자당 평균 10페이지)
- DB 쿼리: 시간당 300,000~600,000 쿼리 (페이지당 5-10 쿼리)
- 쓰기 작업: 분당 100건 (동시접속자의 10% 참여 가정)

## 2. 영역별 검토 결과

**현재 상황:**
- Hobby (무료): 동시 실행 10개, 실행 시간 10초, 100GB-시간

**부하 분석:**

- 평균 응답 시간 1-3초 가정 시 동시 실행 16-50개
- Next.js SSR 평균 렌더링 시간: 0.5-2초

**결론:**
- Hobby 플랜: 동시 실행 10개 제한으로 불가능
- Pro 플랜: 동시 실행 100개로 충분, 실행 시간 60초도 적합
- **권장: Vercel Pro 플랜 이상 필수**

### 💡 비개발자를 위한 최적화 방안 쉬운 설명

#### 1. ISR (15분 캐싱) = 요리 미리 만들어두기

**일상 비유:**
패스트푸드점에서 햄버거를 주문할 때마다 처음부터 만들면 시간이 오래 걸립니다.
그래서 인기 메뉴는 미리 만들어서 보온함에 15분간 보관해두고, 주문이 들어오면 바로 줍니다.
15분 지나면 새로 만들어서 교체합니다.

**웹사이트 적용:**
- 사용자가 "연예 이슈 목록" 페이지를 요청할 때마다 DB에서 데이터 가져오고 페이지 만들면 0.5초 소요
- 대신 15분마다 한 번만 페이지를 만들어두고, 그 사이 방문자들에게는 만들어둔 페이지 제공
- 1,000명이 같은 페이지 보러 와도 한 번만 만들면 되니까 비용 절감

**효과:**
- 속도: 0.5초 → 0.05초 (10배 빠름)
- 비용: 1,000번 만들기 → 1번만 만들기 (1,000배 절감)

#### 2. Edge Caching = 동네마다 편의점 두기

**일상 비유:**
본사 창고가 서울에 있는데, 부산 사람이 물건 주문하면 서울에서 부산까지 배송해야 합니다.
대신 부산 동네마다 편의점을 두고 인기 상품을 미리 배치해두면, 부산 사람은 동네 편의점에서 바로 삽니다.

**웹사이트 적용:**
- 서버: 미국 (Vercel 서버)
- 한국 사용자가 페이지 요청 → 미국까지 왕복 0.3초 소요
- 대신 서울, 부산 등에 복사본 서버(Edge) 두고 거기서 제공 → 0.05초

**효과:**
- 속도: 미국 왕복 0.3초 → 서울 왕복 0.05초
- 사용자 체감: 페이지가 훨씬 빠르게 뜸

#### 3. React Server Components = 짐 가볍게 싸기

**일상 비유:**
여행 갈 때 가방에 옷, 신발, 화장품 다 넣으면 무겁습니다.
대신 현지에서 살 수 있는 건 현지에서 사고, 꼭 필요한 것만 가방에 넣으면 가벼워집니다.

**웹사이트 적용:**
- 기존: 페이지 보여주려면 200KB 파일 다운로드 (JavaScript 코드 전체)
- 개선: 서버에서 미리 처리하고, 사용자는 50KB만 다운로드
- 특히 모바일에서 다운로드 속도 향상

**효과:**
- 다운로드 크기: 200KB → 50KB (4배 감소)
- 모바일 3G: 6초 → 1.5초 (4배 빠름)
- 데이터 요금 절감

#### 4. DB 인덱스 = 책 뒤 찾아보기

**일상 비유:**
500페이지 책에서 "김연경" 단어 찾기:
- 방법1 (인덱스 없음): 1페이지부터 500페이지까지 하나하나 읽기 → 10분 소요
- 방법2 (인덱스 있음): 책 뒤 "찾아보기"에서 "김연경 - 127페이지" 확인 → 10초 소요

**웹사이트 적용:**
- 이슈 테이블에 10,000개 데이터
- "연예 카테고리" 이슈 찾기:
    - 인덱스 없음: 10,000개 다 확인 → 1초
    - 인덱스 있음: 바로 찾기 → 0.01초

**효과:**
- 검색 속도: 1초 → 0.01초 (100배 빠름)
- 동시 접속자 1,000명일 때 필수

### 2.6 네트워크 대역폭

**페이지별 크기 추정:**
- 이슈 목록: HTML 50KB + JS/CSS 200KB = 250KB
- 이슈 상세: HTML 100KB + JS/CSS 200KB = 300KB
- 평균: 270KB/페이지

**대역폭 계산 (캐싱 없음):**
- 시간당 60,000 PV × 270KB = 15.8 GB/시간
- 월간: 15.8 GB × 24 × 30 = 11.1 TB/월

**Vercel 대역폭 비용:**
- Pro 플랜: 1TB 포함, 초과 시 $40/100GB
- 11.1 TB 사용 시: 1TB 포함 + 10.1TB 초과 = $4,040 추가 비용

**Supabase 대역폭:**
- DB 쿼리 응답: 평균 10KB/쿼리
- 월간: 600,000 × 60 × 24 × 30 × 10KB = 247 GB/월
- Pro 플랜: 50GB 포함, 초과 시 $0.09/GB
- 50GB 포함 + 197GB 초과 = $17.73 추가

**문제점:**
- Vercel 대역폭 초과 비용 과다 ($4,040/월)

**결론:**
- **캐싱 없이는 불가능. 적극적인 캐싱 전략 필수**

### 2.7 캐싱 전략 및 최적화

**현재 캐싱:**
- revalidate: 900초 (15분)

**개선 시나리오 1: 캐싱 히트율 70%**
- SSR 처리: 60,000 PV × 30% = 18,000 PV
- 대역폭: 18,000 × 270KB = 4.7 GB/시간
- 월간: 4.7 GB × 24 × 30 = 3.3 TB/월
- Vercel 비용: 1TB 포함 + 2.3TB 초과 = $920 추가

**개선 시나리오 2: 캐싱 히트율 90% (권장)**
- SSR 처리: 60,000 PV × 10% = 6,000 PV
- 대역폭: 6,000 × 270KB = 1.57 GB/시간
- 월간: 1.57 GB × 24 × 30 = 1.1 TB/월
- Vercel 비용: 1TB 포함 + 0.1TB 초과 = $40 추가

**캐싱 전략:**
1. ISR (Incremental Static Regeneration): 주요 이슈 페이지 15분 캐싱
2. CDN Edge Caching: Vercel Edge Network 활용
3. Stale-While-Revalidate: 오래된 콘텐츠 먼저 제공, 백그라운드 갱신
4. Static Pages: 홈, 카테고리 목록 페이지

**결론:**
- **캐싱 히트율 90% 달성 시 비용 적정 수준 ($40-60/월)**
- **적극적인 캐싱 전략 필수**

## 3. DB 인덱스 설계

### 3.1 필수 인덱스

**issues 테이블:**
```sql
CREATE INDEX idx_issues_category_status_heat 
    ON issues(category, status, heat_index DESC);
CREATE INDEX idx_issues_status ON issues(status);
CREATE INDEX idx_issues_created_at ON issues(created_at DESC);
```

**timeline_points 테이블:**
```sql
CREATE INDEX idx_timeline_issue_occurred 
    ON timeline_points(issue_id, occurred_at);
```

**comments 테이블:**
```sql
CREATE INDEX idx_comments_issue_created 
    ON comments(issue_id, created_at DESC);
CREATE INDEX idx_comments_issue_score 
    ON comments(issue_id, score DESC);
```

**reactions 테이블:**
```sql
CREATE INDEX idx_reactions_issue_type 
    ON reactions(issue_id, type);
CREATE UNIQUE INDEX idx_reactions_unique 
    ON reactions(issue_id, user_id);
```

**votes 테이블:**
```sql
CREATE INDEX idx_votes_issue_phase 
    ON votes(issue_id, phase);
```

### 3.2 인덱스 효과

**인덱스 없이:**
- 쿼리 평균 200ms (Full Table Scan)
- 시간당 600,000 쿼리 × 200ms = 33.3시간 (불가능)

**인덱스 적용:**
- 쿼리 평균 20ms (Index Scan)
- 시간당 600,000 쿼리 × 20ms = 3.3시간 (가능)

**결론:**
- **위 5가지 인덱스 반드시 추가 필요**

## 4. Connection Pooling 설정

### 4.1 Supabase Connection Pooler

**모드:**
- Transaction Mode: 트랜잭션마다 연결 생성/해제 (권장)
- Session Mode: 클라이언트당 연결 유지

**설정:**
- Supavisor (Pooler): 최대 3,000개 동시 연결 지원
- 연결 문자열: pooler 포트 (6543) 사용
- max connections: 50-100 (서버리스 함수당)

### 4.2 연결 수 계산

**Pooler 사용 시:**
- 동시 실행 함수: 20-50개 (피크타임)
- 실제 DB 연결: 20-50개
- Supabase Pro 한도 200개로 충분

**Pooler 미사용 시:**
- 동시 실행 함수 50개 × 함수당 5개 연결 = 250개 (한도 초과)

**결론:**
- **Supabase Pooler (Transaction Mode) 필수 사용**
- **미사용 시 연결 한도 초과로 서비스 불가**

### 6.2 필수 조건

**인프라 플랜:**
1. Vercel Pro 플랜 ($20/월) 이상
2. Supabase Pro 플랜 ($25/월) 이상

    - comments: issue_id+created_at, issue_id+score

**성능 최적화:**
5. ISR/Edge Caching 등 캐싱 전략 (히트율 90% 목표)
    - ISR: revalidate 900초 (15분)
    - Edge Caching: Vercel Edge Network
    - Stale-While-Revalidate
    - Static Pages: 홈, 카테고리 목록

**API 관리:**

## 9. 비개발자를 위한 전체 요약

### 9.1 핵심 질문: 우리 서비스 버틸 수 있나?

**질문:** 유저 1만 명, 동시 접속자 1,000명일 때 서비스 가능한가?

**답:** 가능합니다. 단, 준비가 필요합니다.

### 9.2 식당 비유로 이해하기

우리 서비스를 식당으로 비유하면:

#### 현재 상황 (무료 플랜)
- **주방 크기:** 10평 (동시 10명 요리 가능)
- **홀 테이블:** 60개
- **식재료 창고:** 500MB (작음)
- **월 식재료비:** 무료

**문제점:**
- 피크타임에 손님 1,000명 오면 → 대기 줄 엄청 길어짐
- 테이블 60개로 부족 → 손님이 자리 없어서 못 들어옴
- 주방 10평으로 10명만 조리 가능 → 나머지 990명은 대기

#### 개선 후 (Pro 플랜 + 최적화)
- **주방 크기:** 100평 (동시 100명 요리 가능)
- **홀 테이블:** 200개
- **식재료 창고:** 8GB (충분)
- **배달 시스템:** 전국 거점에 배달 센터 (Edge Caching)
- **미리 조리:** 인기 메뉴는 15분마다 미리 만들어둠 (ISR)
- **찾기 시스템:** 식재료 창고에 위치 표지판 (DB 인덱스)

**결과:**
- 손님 1,000명 와도 빠르게 서빙 가능
- 대기 시간: 30초 → 3초
- 월 비용: 13-20만원
