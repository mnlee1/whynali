# 시스템 규모 검토 (유저 10,000명, 동시접속자 1,000명)

> 작성일: 2026-03-13
> 대상 시나리오: 총 유저 10,000명, 피크타임 동시접속자 1,000명

## 1. 검토 개요

왜난리 프로젝트의 시스템 아키텍처가 유저 수 10,000명, 동시접속자 1,000명 규모를 안정적으로 처리할 수 있는지 검토합니다.

### 1.1 현재 기술 스택

- 프론트엔드: Next.js 15.1.6 (App Router), React 19
- 백엔드/DB: Supabase (PostgreSQL)
- 배포: Vercel
- AI: Groq API (무료, 다중 키), Perplexity (선택)
- 크론잡: GitHub Actions / Vercel Cron

### 1.2 트래픽 시나리오

동시접속자 1,000명 기준:
- 평균 세션 시간: 10분
- 시간당 세션: 약 6,000 세션
- 페이지뷰: 시간당 60,000 PV (사용자당 평균 10페이지)
- DB 쿼리: 시간당 300,000~600,000 쿼리 (페이지당 5-10 쿼리)
- 쓰기 작업: 분당 100건 (동시접속자의 10% 참여 가정)

## 2. 영역별 검토 결과

### 2.1 Vercel 서버리스 제약

**현재 상황:**
- Hobby (무료): 동시 실행 10개, 실행 시간 10초, 100GB-시간
- Pro ($20/월): 동시 실행 100개, 실행 시간 60초, 1000GB-시간

**부하 분석:**
- 시간당 60,000 PV = 초당 16.67 요청
- 평균 응답 시간 1-3초 가정 시 동시 실행 16-50개
- Next.js SSR 평균 렌더링 시간: 0.5-2초

**결론:**
- Hobby 플랜: 동시 실행 10개 제한으로 불가능
- Pro 플랜: 동시 실행 100개로 충분, 실행 시간 60초도 적합
- **권장: Vercel Pro 플랜 이상 필수**

### 2.2 Supabase 데이터베이스 성능

**플랜별 제한:**
- Free: 500MB DB, 2GB 대역폭, 연결 60개, 5GB 파일 저장
- Pro ($25/월): 8GB DB, 50GB 대역폭, 연결 200개, 100GB 파일 저장

**부하 분석:**
- 초당 83-167 쿼리 (시간당 300,000~600,000 쿼리)
- PostgreSQL: 적절한 인덱스 시 초당 수천 건 처리 가능
- Connection Pooling: Supavisor 사용 시 3,000개 동시 연결 지원

**연결 관리:**
- 동시 실행 함수 20-50개 (피크타임)
- Pooler 사용 시 실제 DB 연결: 20-50개
- Pooler 미사용 시: 함수당 5개 연결 × 50개 = 250개 (한도 초과)

**결론:**
- Free 플랜: 연결 60개로 불가능
- Pro 플랜: 연결 200개 + Pooling으로 충분
- **권장: Supabase Pro 플랜 + Connection Pooler(Transaction Mode) 필수**

### 2.3 크론잡 부하

**현재 설정:**
- 뉴스 수집: 30분마다 (하루 48회)
- 커뮤니티 수집: 3분마다 (하루 480회)
- 기타: 화력 재계산, 자동 링크, 상태 전환, 투표 자동 종료 등

**GitHub Actions 제한:**
- 무료 플랜: 월 2,000분
- 필요 시간: 뉴스 1,440분 + 커뮤니티 7,200분 = 8,640분/월
- 부족분: 6,640분

**대안:**
1. Vercel Cron Jobs (무료 플랜 제한 적음)
2. GitHub Actions Pro ($4/월): 3,000분 제공 (여전히 부족)
3. 크론 주기 조정: 커뮤니티 3분 → 10분 (4,320분 절감)

**결론:**
- **권장: Vercel Cron Jobs로 전환 또는 크론 주기 조정**

### 2.4 API Rate Limit

**Groq API (무료):**
- RPM: 30, TPM: 6,000, TPD: 14,400
- 현재 사용: 하루 약 150,000 토큰 (TPD 14,400 초과)
- 다중 키: 5개 키 사용 중 (72,000 TPD 가능)

**필요량:**
- 카테고리 분류: 1,000 토큰/일
- 중복 체크: 3,000 토큰/일
- 뉴스 연결 검증: 121,000 토큰/일
- 커뮤니티 매칭: 상위 10개 × 480회 = 추가 부하

**Naver News API:**
- 일일 한도: 25,000건
- 현재 사용: 하루 48회 × 100건 = 4,800건 (19%)

**결론:**
- Groq API: 10-15개 키 필요 (현재 5개에서 2-3배 증가)
- Naver API: 충분
- **권장: Groq API 키 10-15개 준비**

### 2.5 프론트엔드 SSR 부하

**렌더링 시간 추정:**
- 이슈 목록: DB 쿼리 0.1초 + 렌더링 0.2초 = 0.3초
- 이슈 상세: DB 쿼리 0.3초 + 렌더링 0.3초 = 0.6초
- 평균: 0.4-0.5초

**부하 분석:**
- 초당 16.67 요청 × 0.5초 = 동시 실행 8.3개
- 피크 시 (2배): 16-20개 동시 실행

**Vercel Pro 제한:**
- 동시 실행 100개로 충분

**최적화 방안:**
1. ISR (Incremental Static Regeneration): 15분 캐싱
2. Edge Caching: Vercel Edge Network 활용
3. React Server Components: 클라이언트 번들 크기 감소
4. DB 인덱스 최적화

**결론:**
- **SSR 부하는 문제없음. 추가 최적화 권장**

### 2.6 네트워크 대역폭

**페이지별 크기 추정:**
- 이슈 목록: HTML 50KB + JS/CSS 200KB = 250KB
- 이슈 상세: HTML 100KB + JS/CSS 200KB = 300KB
- 평균: 270KB/페이지

**대역폭 계산 (캐싱 없음):**
- 시간당 60,000 PV × 270KB = 15.8 GB/시간
- 월간: 15.8 GB × 24 × 30 = 11.1 TB/월

**Vercel 대역폭 비용:**
- Pro 플랜: 1TB 포함, 초과 시 $40/100GB
- 11.1 TB 사용 시: 1TB 포함 + 10.1TB 초과 = $4,040 추가 비용

**Supabase 대역폭:**
- DB 쿼리 응답: 평균 10KB/쿼리
- 월간: 600,000 × 60 × 24 × 30 × 10KB = 247 GB/월
- Pro 플랜: 50GB 포함, 초과 시 $0.09/GB
- 50GB 포함 + 197GB 초과 = $17.73 추가

**문제점:**
- Vercel 대역폭 초과 비용 과다 ($4,040/월)

**결론:**
- **캐싱 없이는 불가능. 적극적인 캐싱 전략 필수**

### 2.7 캐싱 전략 및 최적화

**현재 캐싱:**
- revalidate: 900초 (15분)

**개선 시나리오 1: 캐싱 히트율 70%**
- SSR 처리: 60,000 PV × 30% = 18,000 PV
- 대역폭: 18,000 × 270KB = 4.7 GB/시간
- 월간: 4.7 GB × 24 × 30 = 3.3 TB/월
- Vercel 비용: 1TB 포함 + 2.3TB 초과 = $920 추가

**개선 시나리오 2: 캐싱 히트율 90% (권장)**
- SSR 처리: 60,000 PV × 10% = 6,000 PV
- 대역폭: 6,000 × 270KB = 1.57 GB/시간
- 월간: 1.57 GB × 24 × 30 = 1.1 TB/월
- Vercel 비용: 1TB 포함 + 0.1TB 초과 = $40 추가

**캐싱 전략:**
1. ISR (Incremental Static Regeneration): 주요 이슈 페이지 15분 캐싱
2. CDN Edge Caching: Vercel Edge Network 활용
3. Stale-While-Revalidate: 오래된 콘텐츠 먼저 제공, 백그라운드 갱신
4. Static Pages: 홈, 카테고리 목록 페이지

**결론:**
- **캐싱 히트율 90% 달성 시 비용 적정 수준 ($40-60/월)**
- **적극적인 캐싱 전략 필수**

## 3. DB 인덱스 설계

### 3.1 필수 인덱스

**issues 테이블:**
```sql
CREATE INDEX idx_issues_category_status_heat 
    ON issues(category, status, heat_index DESC);
CREATE INDEX idx_issues_status ON issues(status);
CREATE INDEX idx_issues_created_at ON issues(created_at DESC);
```

**timeline_points 테이블:**
```sql
CREATE INDEX idx_timeline_issue_occurred 
    ON timeline_points(issue_id, occurred_at);
```

**comments 테이블:**
```sql
CREATE INDEX idx_comments_issue_created 
    ON comments(issue_id, created_at DESC);
CREATE INDEX idx_comments_issue_score 
    ON comments(issue_id, score DESC);
```

**reactions 테이블:**
```sql
CREATE INDEX idx_reactions_issue_type 
    ON reactions(issue_id, type);
CREATE UNIQUE INDEX idx_reactions_unique 
    ON reactions(issue_id, user_id);
```

**votes 테이블:**
```sql
CREATE INDEX idx_votes_issue_phase 
    ON votes(issue_id, phase);
```

### 3.2 인덱스 효과

**인덱스 없이:**
- 쿼리 평균 200ms (Full Table Scan)
- 시간당 600,000 쿼리 × 200ms = 33.3시간 (불가능)

**인덱스 적용:**
- 쿼리 평균 20ms (Index Scan)
- 시간당 600,000 쿼리 × 20ms = 3.3시간 (가능)

**결론:**
- **위 5가지 인덱스 반드시 추가 필요**

## 4. Connection Pooling 설정

### 4.1 Supabase Connection Pooler

**모드:**
- Transaction Mode: 트랜잭션마다 연결 생성/해제 (권장)
- Session Mode: 클라이언트당 연결 유지

**설정:**
- Supavisor (Pooler): 최대 3,000개 동시 연결 지원
- 연결 문자열: pooler 포트 (6543) 사용
- max connections: 50-100 (서버리스 함수당)

### 4.2 연결 수 계산

**Pooler 사용 시:**
- 동시 실행 함수: 20-50개 (피크타임)
- 실제 DB 연결: 20-50개
- Supabase Pro 한도 200개로 충분

**Pooler 미사용 시:**
- 동시 실행 함수 50개 × 함수당 5개 연결 = 250개 (한도 초과)

**결론:**
- **Supabase Pooler (Transaction Mode) 필수 사용**
- **미사용 시 연결 한도 초과로 서비스 불가**
