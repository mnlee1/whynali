# 커뮤니티 수집 데이터 업데이트 문제 해결

## 문제 원인

1. **`ignoreDuplicates: true` 옵션 사용**: 기존 코드에서 `upsert` 시 `ignoreDuplicates: true` 옵션을 사용하여 중복 URL 발견 시 데이터를 업데이트하지 않고 무시했습니다.

2. **`updated_at` 컬럼 부재**: `community_data` 테이블에 `updated_at` 컬럼이 없어서 마지막 업데이트 시각을 추적할 수 없었습니다.

3. **네이트판 `written_at` 덮어쓰기**: 네이트판은 작성일 정보를 제공하지 않아, 매번 수집 시 현재 시각으로 `written_at`을 덮어써서 원본 작성일이 유실되었습니다.

4. **메인 페이지 노출 게시글만 수집**: 더쿠/네이트판 메인 페이지에 보이는 게시글만 수집하므로, 페이지에서 사라진 인기 게시글은 더 이상 업데이트되지 않음.

## 해결 방안

### 1. 코드 수정 완료

#### lib/collectors/community.ts
- `ignoreDuplicates: true` 옵션 제거
- `updated_at` 필드 추가하여 매 수집 시 현재 시각 기록
- 네이트판: 기존 데이터가 있으면 `written_at` 유지, 없으면 현재 시각 사용
- **이슈 연결 게시글 추가 크롤링**: 최근 48시간 이내 작성되고 이슈에 연결된 게시글은 메인 페이지에 없어도 개별 페이지를 직접 크롤링하여 조회수/댓글수 업데이트 (최대 10개)

#### app/admin/collections/page.tsx
- `updated_at` 컬럼 표시 추가
- 정렬 옵션에 `updated_at` 추가

#### app/api/admin/collections/community/route.ts
- `updated_at` 필드 조회 및 정렬 지원

### 2. DB 마이그레이션 필요

아래 SQL을 Supabase SQL Editor에서 실행하세요:

```sql
-- community_data에 updated_at 추가
ALTER TABLE community_data
    ADD COLUMN IF NOT EXISTS updated_at TIMESTAMPTZ DEFAULT NOW();

-- news_data에 updated_at 추가 (일관성 유지)
ALTER TABLE news_data
    ADD COLUMN IF NOT EXISTS updated_at TIMESTAMPTZ DEFAULT NOW();

-- 기존 레코드의 updated_at을 created_at으로 초기화
UPDATE community_data SET updated_at = created_at WHERE updated_at IS NULL;
UPDATE news_data SET updated_at = created_at WHERE updated_at IS NULL;

-- updated_at 인덱스 추가 (정렬 성능 향상)
CREATE INDEX IF NOT EXISTS idx_community_data_updated_at ON community_data(updated_at);
CREATE INDEX IF NOT EXISTS idx_news_data_updated_at ON news_data(updated_at);
```

### 3. 마이그레이션 실행 방법

1. Supabase Dashboard 접속
2. SQL Editor 메뉴 선택
3. 위 SQL 전체 복사 후 붙여넣기
4. "Run" 버튼 클릭
5. 오류 없이 완료되면 성공

### 4. 확인 방법

마이그레이션 실행 후:

1. 3분 주기 Cron이 실행되면 자동으로 `updated_at`이 갱신됨
2. 관리자 > 수집 현황 페이지에서 "갱신일" 컬럼 확인
3. "갱신일" 기준으로 정렬하여 최근 업데이트된 데이터 확인
4. 같은 URL의 게시글이 수집될 때마다 조회수, 댓글수, 갱신일이 업데이트됨
5. **이슈에 연결된 게시글**은 메인 페이지에서 사라져도 계속 업데이트됨 (48시간 이내 작성 기준)

## 최종 해결 완료

### 수정 내역

1. **더쿠 댓글수 파싱 수정** ✅
   - 정규식: `\d{1,3}(?:,\d{3})*` → `\d+(?:,\d{3})*`
   - "1179"가 "117"과 "9"로 분리되던 문제 해결
   - 모든 자릿수의 숫자 정확히 매칭

2. **네이트판 수집 확인** ✅
   - 조회수, 댓글수 정상 파싱 확인
   - 매 수집 시 50건 정상 수집

### 테스트 결과 (민희진 게시글)
- ✅ 조회수: 59,155 (정확!)
- ✅ 댓글수: 1,179 (정확!)
- ✅ 화력 차이: 45.5점 → 80점 (76% 증가)

### 화력 지수 영향
댓글수가 화력 계산에서 **45% 가중치**를 차지하므로, 정확한 댓글수 수집이 이슈 화력 산정에 결정적입니다.

### 문제
더쿠의 당일 게시글 시간(예: "23:50")을 파싱할 때, 자정 이후 수집하면 미래 시각으로 잘못 파싱되는 문제가 있었습니다.

**예시:**
- 실제 시각: 2026-02-27 00:30 (자정 넘어서 수집)
- 더쿠 표시: "23:50" (전날 밤 11시 50분 작성)
- 잘못된 파싱: 2026-02-27 23:50 (미래!)
- 결과: **작성일 > 수집일** (논리적으로 불가능)

### 해결
`parseTheqooTime` 함수에서 파싱된 시각이 현재보다 미래면 **하루 빼서 전날로 처리**하도록 수정했습니다.

```typescript
/* 파싱된 시각이 현재보다 미래면 전날 게시글 (자정 넘어서 수집한 경우) */
if (d > now) {
    d.setDate(d.getDate() - 1)
}
```

이제 작성일이 항상 수집일보다 이전이거나 같은 시각으로 정확하게 기록됩니다.

## 이슈 연결 게시글 추가 크롤링

### 배경
메인 페이지에서 사라진 게시글은 더 이상 업데이트되지 않아, 이슈에 연결된 중요한 게시글의 조회수/댓글수가 오래된 데이터로 고정되는 문제가 있었습니다.

### 해결
최근 48시간 이내 작성되고 이슈에 연결된 게시글은:
1. 메인 페이지 수집 후 DB에서 이슈 연결 게시글 조회
2. 메인 페이지에 없는 게시글 URL 추출
3. 각 게시글 개별 페이지를 직접 크롤링 (최대 10개)
4. 최신 조회수/댓글수로 DB 업데이트

### 결과
- 이슈에 연결된 중요 게시글은 계속 추적됨
- 실시간 조회수/댓글수 업데이트 유지
- 성능 고려: 48시간 이내 + 최대 10개 제한
