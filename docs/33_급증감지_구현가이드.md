## 커뮤니티 급증 감지 (Track 2)

### 동작 흐름

```
1. 최근 10분간 커뮤니티 글 수집
   ↓
2. 키워드 추출 및 빈도 계산
   ↓
3. 급증 키워드 필터링 (10건 이상)
   ↓
4. AI 이슈 검증 (Groq)
   - 신뢰도 70% 이상
   - 밈/드립/스팸 필터링
   ↓
5. 네이버 뉴스 검색
   - 최소 3건 이상
   ↓
6. 중복 체크
   ↓
7. 긴급 이슈 생성
```

### AI 이슈 검증

**파일**: `lib/candidate/community-burst-detector.ts`

```typescript
// 검증 기준
1. 사회적 논란/사건 → 이슈 O
2. 단순 유행어/밈 → 이슈 X
3. 스팸/홍보 → 이슈 X
4. 정보 가치 없는 반응 → 이슈 X

// 예시
"민희진 논란" → ✅ 이슈 (신뢰도 95%)
"ㅋㅋㅋ" → ❌ 이슈 X (신뢰도 15%)
"띵작" → ⚠️ 이슈 O (신뢰도 72%) but 뉴스 부족
```

### 환경변수

```bash
ENABLE_COMMUNITY_BURST=true           # 활성화
COMMUNITY_BURST_THRESHOLD=10          # 10분간 10건
COMMUNITY_BURST_WINDOW_MINUTES=10     # 시간 창
```

## AI 중복 체크

### 4단계 검증

**파일**: `lib/candidate/duplicate-checker.ts`

```
1단계: 정확한 제목 일치
  → "민희진 논란" = "민희진 논란" (중복)

2단계: 키워드 필터링
  → 공통 키워드 2개 이상만 AI 체크
  
3단계: 안전 장치
  → 반대어: "복귀" vs "사퇴" (별개)
  → 숫자 차이: "1차 회견" vs "2차 회견" (별개)
  
4단계: AI 정밀 비교
  → Groq AI로 신뢰도 80% 이상 중복 판단
```

### 테스트 결과

```
반대어 감지:
✅ "민희진 복귀" vs "민희진 사퇴" → 별개
✅ "이재명 기소" vs "이재명 무혐의" → 별개

숫자 차이 감지:
✅ "민희진 1차 회견" vs "민희진 2차 회견" → 별개
✅ "이재명 재판 3일차" vs "4일차" → 별개
```

### 환경변수

```bash
DUPLICATE_CHECK_AI_CONFIDENCE=80      # 신뢰도 (80%)
DUPLICATE_CHECK_WINDOW_HOURS=1        # 시간 창 (1시간)
```

## 비용 상세

### AI 호출 빈도

**중복 체크** (lib/candidate/duplicate-checker.ts):
- 호출 시점: 새 이슈 생성 시
- 최대 호출: 3회/이슈
- 예상: 시간당 5개 이슈 = 15회

**커뮤니티 검증** (lib/candidate/community-burst-detector.ts):
- 호출 시점: 3분마다
- 최대 호출: 3개 키워드 × 20회/시간 = 60회

**합계**: 시간당 75회 = 일일 1,800회

### 무료 한도

**Groq AI**:
- 무료 한도: 14,400회/일
- 사용량: 1,800회/일 (12.5%)
- 여유: 12,600회/일 (87.5%)

**GitHub Actions**:
- 무료 한도: 2,000분/월 (Private)
- 사용량: ~240분/월
- 여유: 충분

## 모니터링 쿼리

### 급증 이슈 통계

```sql
-- 오늘 생성된 급증 이슈
SELECT 
    source_track,
    COUNT(*) as count,
    AVG(burst_level) as avg_level,
    AVG(heat_index) as avg_heat
FROM issues
WHERE is_urgent = true
  AND created_at >= CURRENT_DATE
GROUP BY source_track;

-- 시간대별 급증 이슈
SELECT 
    DATE_TRUNC('hour', created_at) as hour,
    source_track,
    COUNT(*) as count,
    AVG(burst_level) as avg_level
FROM issues
WHERE is_urgent = true
  AND created_at >= CURRENT_DATE - INTERVAL '1 day'
GROUP BY hour, source_track
ORDER BY hour DESC;

-- 트랙별 승인율
SELECT 
    source_track,
    approval_status,
    COUNT(*) as count,
    ROUND(COUNT(*) * 100.0 / SUM(COUNT(*)) OVER(PARTITION BY source_track), 1) as percentage
FROM issues
WHERE is_urgent = true
GROUP BY source_track, approval_status
ORDER BY source_track, approval_status;
```
