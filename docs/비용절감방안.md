### Fallback 시스템 현황
현재 `AI_PROVIDER=claude-fallback` 설정으로 **이미 자동 전환이 작동 중**:
- 4월 12일: Claude 183번 전부 실패 → Groq 223번 사용 (218번 성공)
- FallbackProvider가 크레딧 부족 감지 시 자동으로 Groq 전환

**문제점**:
1. 매번 Claude를 먼저 시도하고 실패한 후 Groq으로 전환
2. 불필요한 실패 호출로 API 통계 오염
3. Claude 실패 시간만큼 응답 지연 발생 (약 1~2초)

### 비용 추정

- 하지만 Claude 재충전 시 다시 비용 발생

### Vercel 빌드 비용
- ignoreCommand 설정으로 이미 최적화됨 (main 브랜치만 빌드)
- 추가 최적화 불필요

## 비용 절감 방안

### 방안 1: AI 프로바이더를 Groq으로 직접 전환 (권장)

**현재 상황**:
- ✅ Fallback이 작동 중: Claude 실패 → Groq 자동 전환
- ⚠️ 하지만 매번 Claude를 먼저 시도하고 실패 후 Groq으로 전환
- ⚠️ 불필요한 지연 (1~2초) 및 API 실패 기록 누적

**효과**: 
- 추가 비용 없음 (이미 Groq 사용 중)
- 응답 속도 개선 (Claude 실패 대기 시간 제거)
- API 통계 정리 (실패 건수 제거)
- Claude 크레딧 재충전 시 비용 재발 방지

**장점**:
- Groq은 완전 무료
- 현재 성공률 95.9%로 안정적
- 7개 API 키 로테이션으로 Rate Limit 회피 중
- 품질: Llama 3.1 8B 모델로 토론/투표 생성에 충분

**적용 방법**:

1. Vercel 환경 변수 변경:
```bash
AI_PROVIDER=groq
```

2. 또는 claude-fallback 유지 + ANTHROPIC_API_KEY 제거 (Claude 시도 자체를 막음)

**권장**: 옵션 1 (`AI_PROVIDER=groq`)로 명확하게 설정

### 방안 2: AI 호출 빈도 최적화 (중장기 효과)

**현재 AI 사용 지점**:
1. 매일 새벽 3시 크론 (daily-generate-content)
   - 토론 주제: 최대 15개/일 (이슈 5개 × 3개)
   - 투표: 최대 5개/일
   - 숏폼 이미지 검증: 변동
2. 관리자 수동 생성
3. 사용자 요청 기반 생성

**최적화 방안**:

#### 2-1. 일일 생성 이슈 수 제한
현재 설정: `MAX_ISSUES_PER_RUN=5`

```env
# 하루 처리 이슈 수를 3개로 축소
DAILY_GENERATE_MAX_ISSUES=3
```

**효과**: AI 호출 약 40% 감소 (토론 15개 → 9개, 투표 5개 → 3개)

#### 2-2. 토론 주제 생성 개수 조정
현재: 이슈당 3개 생성

```typescript
// lib/ai/discussion-generator.ts 수정
const topics = await generateDiscussionTopics(metadata, 2) // 3 → 2
```

**효과**: 토론 생성 33% 감소

#### 2-3. 화력 지수 임계값 상향
현재: heat_index ≥ 15

```env
# 화력 높은 이슈만 처리
DAILY_GENERATE_MIN_HEAT=30
```

**효과**: 처리 대상 이슈 약 50% 감소 (예상)

### 방안 3: Claude 크레딧 충전 (비용 증가)

Claude API를 계속 사용하려면:
- 월 $60 크레딧 충전 필요
- 또는 토큰 사용량 모니터링 강화

**비권장**: Groq 무료 사용이 더 효율적

### 방안 4: 하이브리드 전략

특정 기능만 Claude 사용:
```typescript
// 중요한 기능(관리자 수동 생성)만 Claude
// 자동 배치는 Groq
```

구현 복잡도가 높아 비권장

## 추천 실행 계획

1. Vercel 환경 변수 변경: `AI_PROVIDER=groq`
2. 변경 후 배포 확인
3. API 사용 현황 모니터링 (1주일)

**예상 절감**: Claude API $45~60 → $0

### 선택적 실행 (2주 내)
Groq 품질이 충분하다면 추가 최적화 불필요.
만약 Groq도 Rate Limit에 걸린다면:

1. 일일 생성 이슈 수 축소: `DAILY_GENERATE_MAX_ISSUES=3`
2. 화력 임계값 상향: `DAILY_GENERATE_MIN_HEAT=20`

**예상 절감**: 없음 (Groq은 무료이므로)

### 모니터링 지표
```bash
npm run check:api
```

주간 확인 사항:
- Groq 성공률 (목표: 95% 이상 유지)
- Groq Rate Limit 발생 여부
- 토론/투표 품질 (사용자 피드백)

- ⚠️ 응답 지연 (1~2초) 및 API 실패 통계 누적

- Vercel에서 `AI_PROVIDER=groq`으로 변경
- Claude 시도를 건너뛰고 바로 Groq 사용
- **추가 비용 절감 효과는 없지만** (이미 Groq 사용 중):
  - 응답 속도 개선 (1~2초)
