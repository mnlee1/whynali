# 온보딩 플로우 구현 완료

작성일: 2026-03-16
상태: 구현 완료, 버그 수정 완료

## 개요

신규 유저가 소셜 로그인 후 약관 동의와 닉네임 설정을 완료하는 온보딩 플로우 구현. 실명 노출 방지 및 안정성 개선 완료.

---

## 구현 파일

### 1. DB 마이그레이션 (3개)

#### A. terms_agreed_at 컬럼 추가

파일: `supabase/migrations/add_terms_columns_to_users.sql`

```sql
ALTER TABLE users ADD COLUMN IF NOT EXISTS terms_agreed_at TIMESTAMPTZ;
ALTER TABLE users ADD COLUMN IF NOT EXISTS marketing_agreed BOOLEAN DEFAULT FALSE;
CREATE INDEX IF NOT EXISTS idx_users_terms_agreed_at ON users(terms_agreed_at);
```

**역할:**
- terms_agreed_at IS NULL = 신규 유저 (온보딩 필요)
- terms_agreed_at NOT NULL = 온보딩 완료 유저

#### B. Auth 트리거 생성

파일: `supabase/migrations/create_users_on_auth_signup.sql`

**역할:**
- Auth 유저 가입 시 users 테이블 자동 생성
- Google, Kakao, Naver 모든 소셜 로그인 지원

#### C. display_name NULL 설정 (실명 노출 방지)

파일: `supabase/migrations/fix_users_trigger_no_display_name.sql`

```sql
INSERT INTO public.users (id, provider, provider_id, display_name)
VALUES (
    NEW.id,
    provider_name,
    provider_user_id,
    NULL  -- 온보딩에서 랜덤 닉네임으로 설정
)
```

**역할:**
- 신규 유저 display_name을 NULL로 설정
- 온보딩 완료 전까지 실명 노출 방지

### 2. 랜덤 닉네임 생성

파일: `lib/random-nickname.ts`

**기능:**
- 형식: 형용사 + 동물 + 4자리숫자
- `generateRandomNickname()`: 랜덤 닉네임 생성
- `generateUniqueNickname(supabase)`: DB 중복 확인 (최대 5회 시도)
- 형용사 25개, 동물 25개 풀 제공

**예시:**
```
행복한토끼1234
용감한사자5678
귀여운강아지9012
```

```

5. 온보딩 완료 후 메인 페이지로 이동
```

## 디버그 가이드

### 온보딩이 작동하지 않는 경우

#### 1단계: 현재 유저 상태 확인

로그인 후 브라우저에서 접속:
```
http://localhost:3000/api/debug/check-user
```

**정상 결과:**
```json
{
    "authenticated": true,
    "authUser": {
        "id": "uuid",
        "email": "test@gmail.com",
        "provider": "google"
    },
    "dbUser": {
        "id": "uuid",
        "provider": "구글",
        "display_name": null,
        "terms_agreed_at": null,
        "marketing_agreed": false,
        "created_at": "2026-03-16T..."
    },
    "needsOnboarding": true
}
```

#### 2단계: 문제 케이스 진단

**케이스 1: terms_agreed_at에 값이 있음**
```json
"terms_agreed_at": "2026-03-16T10:00:00Z"
```
→ 이미 온보딩 완료 (정상)

**케이스 2: dbUser가 null**
```json
"dbUser": null,
"dbError": "No rows found"
```
→ users 테이블에 레코드 없음
→ 트리거가 작동하지 않음
→ 마이그레이션 B 재실행

**케이스 3: terms_agreed_at 컬럼이 없음**
```json
"dbError": "column \"terms_agreed_at\" does not exist"
```
→ 마이그레이션 A 실행 필요

#### 3단계: 로그 확인

**서버 로그 (로그인 시):**
```
[auth/callback] 유저 로그인: uuid test@gmail.com
[auth/callback] DB 유저 조회: {
    userId: "uuid",
    userData: { terms_agreed_at: null },
    needsOnboarding: true
}
[auth/callback] → /onboarding 리다이렉트
```

로컬 테스트:
```bash
npm run dev
```

터미널에서 위 로그 확인

#### 4단계: 강제 온보딩 테스트

기존 유저를 신규 유저로 변경:

```sql
-- 방법 1: 이메일로 ID 찾기
SELECT id FROM auth.users WHERE email = 'test@naver.com';

-- 위에서 얻은 ID로 업데이트
UPDATE users 
SET terms_agreed_at = NULL, display_name = NULL 
WHERE id = 'UUID-붙여넣기';

-- 방법 2: 한번에 실행
UPDATE users 
SET terms_agreed_at = NULL, display_name = NULL 
WHERE id = (SELECT id FROM auth.users WHERE email = 'test@naver.com');
```

재로그인하면 온보딩 페이지로 이동해야 함

### 전체 유저 온보딩 상태 확인

```sql
SELECT 
    u.id,
    au.email,
    u.provider,
    u.display_name,
    u.terms_agreed_at,
    CASE 
        WHEN u.terms_agreed_at IS NULL THEN '온보딩 필요'
        ELSE '온보딩 완료'
    END as status,
    u.created_at
FROM users u
LEFT JOIN auth.users au ON u.id = au.id
ORDER BY u.created_at DESC
LIMIT 10;
```
