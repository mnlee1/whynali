# Rate Limit 해결 방안 가이드

## 문제 상황

```
에러 메시지:
모든 Groq API 키가 Rate Limit 상태입니다. 잠시 후 다시 시도해주세요.

근본 원인:
- Groq API 무료 티어: 분당 ~30 RPM, 일일 ~14,400 RPD
- Track A 한 번 실행: 3개 키워드 × 4-5회 AI 호출 = 12-15 요청
- 크론 주기: 30분마다 실행 시 일일 576-720 요청
```

## 적용된 완화 조치 (현재 구현)

### 1. 처리량 감소
**설정 파일**: `.env.local`

```bash
# 한 번에 1개 키워드만 처리 (기존 3개)
TRACK_A_MAX_KEYWORDS=1

# AI 호출 간 10초 대기 (기존 3초)
TRACK_A_AI_DELAY_MS=10000
```

**효과**: 일일 요청 수를 ~200-240개로 감소 (약 70% 절감)

### 2. 지수 백오프 재시도
**파일**: `lib/ai/groq-provider.ts`

Rate Limit 발생 시:
- 1회: 2초 대기
- 2회: 4초 대기
- 3회: 8초 대기

### 3. 조기 종료
**파일**: `app/api/cron/track-a/route.ts`

모든 키가 Rate Limit 상태면 즉시 크론 종료하여 불필요한 재시도 방지.

## 추가 조정 가능 옵션

### 옵션 A: 크론 주기 늘리기
**현재**: 30분마다 실행
**제안**: 60분마다 실행

```yaml
# .github/workflows/cron-track-a.yml
schedule:
    - cron: '0 * * * *'  # 매시 정각 (기존 30분 주기에서 변경)
```

**효과**: 일일 요청 수를 절반으로 감소

### 옵션 B: 키워드 수 더 줄이기
```bash
# .env.local
TRACK_A_MAX_KEYWORDS=1        # 이미 1개
TRACK_A_AI_DELAY_MS=15000     # 15초로 증가 (기존 10초)
```

### 옵션 C: 비활성 시간대 설정
Track A를 특정 시간대에만 실행 (예: 오전 9시-오후 9시)

```yaml
# .github/workflows/cron-track-a.yml
schedule:
    - cron: '0 9-21/2 * * *'  # 오전 9시부터 오후 9시까지 2시간마다
```

## 근본적 해결책: 유료 플랜 업그레이드

### Groq Pay-As-You-Go (추천)
**가격**: $0.05 ~ $0.59 / 1M 토큰 (모델별 상이)
**제한**: 분당 6,000 RPM, 일일 14,400 RPD

**예상 비용 (Track A 기준)**:
- 일일 요청: ~720회
- 평균 토큰: ~500 토큰/요청
- 총 토큰: 360,000 토큰/일 = 10.8M 토큰/월
- **월 비용**: 약 $0.54 ~ $6.37 (모델별)

**가입 방법**:
1. https://console.groq.com/settings/billing 접속
2. 결제 수단 등록 (신용카드)
3. Pay-As-You-Go 활성화
4. 기존 API 키 그대로 사용 가능 (자동 업그레이드)

### Groq Enterprise (고성능)
**제한**: 분당 30,000 RPM (무제한에 가까움)
**문의**: https://groq.com/contact

## 결론

**현재 상태**:
- 완화 조치 적용으로 Rate Limit 빈도 ~70% 감소
- 무료 티어에서 최대한 효율화

**권장 사항**:
1. 현재 설정으로 1-2일 모니터링
2. 여전히 Rate Limit 발생 시 → 유료 플랜 업그레이드 ($0.54/월 수준)
3. 서비스 확장 시 → Enterprise 고려

**비용 대비 효과**:
- 무료 티어: 제약 많음, 운영 불안정
- 유료 티어: 월 $1 미만으로 안정적 운영 가능
- **투자 대비 효과 매우 높음** (서비스 안정성 확보)
