# 운영 로그 기록 안 되는 문제 수정

## 문제 상황

`admin_logs` 테이블에 로그가 하나도 기록되지 않는 문제 발견.

## 원인 분석

**데이터베이스 스키마와 애플리케이션 코드 간 타입 불일치:**

- **DB 스키마**: `admin_id UUID REFERENCES users(id)` (UUID 타입)
- **애플리케이션 코드**: `adminEmail` (TEXT 타입, 이메일 주소)

`lib/admin-log.ts`의 `writeAdminLog()` 함수가 이메일 주소(string)를 UUID 컬럼에 insert하려고 시도하여 에러 발생. 함수 내부에서 에러를 silent하게 처리(`catch { /* intentionally silent */ }`)하므로 로그 기록 실패가 외부로 드러나지 않았음.

## 해결 방법

### 1. 마이그레이션 파일 생성

`supabase/migrations/fix_admin_logs_admin_id_type.sql` 생성:

- `admin_id` 컬럼 타입 변경: UUID → TEXT
- `users` 테이블 FK 제약 조건 제거
- 성능 향상을 위한 인덱스 3개 추가:
  - `idx_admin_logs_created_at` (시간순 조회)
  - `idx_admin_logs_admin_id` (관리자별 필터링)
  - `idx_admin_logs_target_type` (대상 유형별 필터링)

### 2. 스키마 파일 업데이트

`supabase/schema.sql` 업데이트:

- `admin_logs` 테이블 정의에서 `admin_id` 타입 수정
- `details TEXT` 컬럼 추가 (이미 마이그레이션에서 추가됨)
- 인덱스 3개 추가

## 참고

`lib/admin-log.ts`의 `writeAdminLog()` 함수는 에러를 silent하게 처리하므로, 마이그레이션 적용 전에는 로그 기록 실패가 외부로 드러나지 않습니다. 이는 의도된 동작으로, 로그 기록 실패가 메인 기능(이슈 승인, 댓글 삭제 등)을 막지 않도록 하기 위함입니다.
