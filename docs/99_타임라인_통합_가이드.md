## 1. 시스템 개요

### 타임라인 시스템이란

이슈 상세 페이지의 타임라인 시스템. 이슈의 진행 과정을 발단/전개/파생/진정 4단계로 분류하고, ChatGPT 스타일의 bullet points로 요약합니다.

### 주요 특징

- ChatGPT 스타일 bullet points 표시
- AI 자동 요약 (Groq API)
- 뉴스 제목만 사용 (본문 수집 안 함)
- 법적 안전성 확보
- 자동 업데이트 (cron)

### 완료된 작업

- 타임라인 표시 문제 해결
- AI 프롬프트 개선 (v2 - 품질 우선)
- ChatGPT 스타일 구현 (bullets)
- 법적 안전성 검증
- 한줄요약 섹션 제거

## 2. DB 스키마 및 구조

### timeline_summaries 테이블

```sql
CREATE TABLE timeline_summaries (
    id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    issue_id UUID NOT NULL REFERENCES issues(id),
    stage TEXT NOT NULL CHECK (stage IN ('발단', '전개', '파생', '진정')),
    stage_title TEXT NOT NULL,
    bullets JSONB NOT NULL DEFAULT '[]'::jsonb,
    summary TEXT,
    date_start TIMESTAMPTZ,
    date_end TIMESTAMPTZ,
    generated_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

### timeline_points 테이블

```sql
CREATE TABLE timeline_points (
    id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    issue_id UUID NOT NULL REFERENCES issues(id),
    stage TEXT NOT NULL CHECK (stage IN ('발단', '전개', '파생', '진정')),
    title TEXT NOT NULL,
    source_url TEXT NOT NULL,
    occurred_at TIMESTAMPTZ NOT NULL,
    ai_summary TEXT,
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    UNIQUE (issue_id, source_url)
);
```

```
- 위에 제공된 기사 제목에 있는 내용만 사용하세요.
- 기사 본문은 수집되지 않았으므로, 제목에 없는 내용은 절대 추가하지 마세요.
- 이는 저작권법 준수를 위한 필수 조건입니다.

## 작업: Bullet Points 요약
각 단계에 대해 아래 규칙을 엄격히 지켜 요약해주세요:

1. **제목에 있는 사실만 사용**: 없는 내용은 절대 추가하지 마세요.
2. **간결하게 작성**: 불필요하게 내용을 늘리지 마세요. 핵심만 전달하세요.
3. **Bullet Points 형태**:
   - 기사가 1개: 1개 bullet point
   - 기사가 2~3개: 2~3개 bullet points로 분리
   - 기사가 4개 이상: 3~5개 bullet points로 주제별 그룹핑
4. **각 bullet은 한 문장**: 간결하게 핵심만 전달
5. **완화 표현 사용**: 확실하지 않은 내용은 "~한 것으로 알려짐", "~로 보임" 등으로 표현
6. **stageTitle**: 10자 이내로 핵심만 추출

품질 우선: 인위적으로 문장을 늘리지 말고, 제목에서 파악되는 핵심만 명확하게 전달하세요.

반드시 아래 JSON 형식으로만 답하세요:
{
  "summaries": [{"stage":"발단","stageTitle":"제목","bullets":["사건1","사건2"]}]
}
```

### 프롬프트 설계 원칙

1. 품질 > 분량
2. 간결함 우선
3. 자연스러운 흐름
4. 내용 늘리기 금지
5. 핵심만 전달

## 4. 개별 포인트 AI 요약

### 개요

타임라인의 각 포인트를 "타이틀: 설명" 형식으로 표시합니다.

예시:
```
[발단]
촬영 현장 사고 (2026년 4월 10일)
서울 OO동 드라마 촬영장에서 스태프 사망 사고 발생

[전개]
경찰 수사 착수 (2시간 전)
경찰이 안전 관리 소홀 여부 조사 시작
```

### 날짜 표기

- 3일 이내: "방금 전", "2시간 전", "1일 전"
- 3일 초과: "2026년 4월 10일"
- 기준: 뉴스 발행 시각 (published_at)

### 데이터 흐름

신규 이슈 생성 (track-a):
1. 뉴스 5개 선별
2. Groq 1회 호출로 단계 분류 + 개별 요약 + 단계별 요약 + 브리핑 생성
3. timeline_points에 ai_summary 포함하여 저장

기존 이슈 업데이트 (update-timeline):
1. 새 뉴스 감지
2. Groq으로 전개/파생 분류 + AI 요약 생성
3. 기존 타임라인과 키워드 중복 체크
4. timeline_points에 추가

사용자 조회:
1. /api/issues/[id]/timeline 호출
2. timeline_points.ai_summary 표시
3. ai_summary가 없으면 원본 title 표시 (호환성)

### AI 프롬프트 (개별 포인트)

파일: `app/api/cron/track-a/route.ts`, `app/api/cron/update-timeline/route.ts`

```
각 뉴스를 "타이틀: 설명" 형식으로 요약하되,
- 중복된 상황 변화는 하나만 선택하고 나머지는 제외
- 타이틀은 3~5단어, 설명은 1문장
- 제목에 없는 내용 추측 금지
```

1차: URL 중복 체크

```typescript
    // 핵심 키워드 3개 이상 겹치면 중복

```

3차: AI 중복 필터링

## 5. 법적 안전성

### 결론: 법적으로 매우 안전함

### 핵심 안전장치

1. 뉴스 본문 수집 안 함 - 저작권 침해 위험 없음
2. 제목만 AI에 전송 - 사실 정보만 사용
3. AI 요약은 우리 창작물 - 원저작물 복제 아님
4. 면책 문구 표시 - "AI가 뉴스 제목을 바탕으로 자동 생성"
5. 출처 링크 제공 - 언론사 트래픽 기여

### 유리한 판례

- 대법원 2012다76829: "뉴스 제목을 링크와 함께 표시하는 것은 저작권 침해가 아니다"
- 대법원 2008다55354: "공개된 정보를 수집하여 분석한 결과는 수집자의 저작물이다"
- 대법원 2010da87777: "정당하게 제공받은 정보를 약관 범위 내에서 사용하는 것은 저작권 침해가 아니다"

### 데이터 처리 흐름

```
네이버 뉴스 API 호출
  ↓
제목, 링크, 출처, 날짜만 수집 (본문 수집 안 함)
  ↓
news_data 테이블 저장
  ↓
timeline_points 조회 (이슈별 뉴스 제목 목록)
  ↓
뉴스 제목만 Groq AI에 전송 (본문 전송 안 함)
  ↓
AI가 제목 기반으로 bullet points 생성
  ↓
timeline_summaries 테이블 저장
  ↓
사용자에게 표시 (면책 문구 포함)
```

### 법적 위험도: 매우 낮음

근거:
- 뉴스 본문 수집 안 함
- 제목만 사용 (사실 정보)
- AI 생성은 우리 창작물
- 유리한 판례 다수
- 유사 서비스 합법 운영 중 (Google News, Perplexity AI, ChatGPT)
- 면책 문구 표시
- 출처 링크 제공

### DB 확인

```sql
-- 요약 생성 현황
SELECT 
    COUNT(DISTINCT issue_id) as issue_count,
    COUNT(*) as summary_count
FROM timeline_summaries;

-- 최근 생성된 요약 확인
SELECT 
    i.title,
    ts.stage,
    ts.stage_title,
    ts.bullets,
    jsonb_array_length(ts.bullets) as bullet_count,
    ts.generated_at
FROM timeline_summaries ts
JOIN issues i ON ts.issue_id = i.id
ORDER BY ts.generated_at DESC
LIMIT 10;

-- bullet 개수 분포
SELECT 
    stage,
    AVG(jsonb_array_length(bullets)) as avg_bullets,
    MIN(jsonb_array_length(bullets)) as min_bullets,
    MAX(jsonb_array_length(bullets)) as max_bullets
FROM timeline_summaries
GROUP BY stage;
```

- 법적 안전성: 제목에 없는 내용을 추측하지 않았는가?
- 완화 표현: 불확실한 내용은 적절히 표현했는가?

## 8. 문제 해결

### 요약이 생성되지 않는 경우

원인 1: timeline_points가 없음
```sql
-- 확인
SELECT COUNT(*) FROM timeline_points WHERE issue_id = '<issue-id>';

-- 해결: update-timeline cron 실행
curl http://localhost:3000/api/cron/update-timeline
```

원인 2: Groq API 에러
```
서버 로그 확인:
⚠️ [요약 생성 실패] <이슈명>: Rate limit exceeded

해결:
- 800ms 대기 시간 증가
- Groq API 키 순환 확인
- 다시 실행
```

원인 3: DB 연결 오류
```
서버 로그 확인:
⚠️ [요약 저장 실패] <이슈명>: connection timeout

해결:
- Supabase 연결 확인
- 다시 실행
```

### 요약이 너무 짧은 경우

원인: 이전 프롬프트로 생성됨

해결:
```javascript
// force=true로 전체 재생성
fetch('/api/admin/migrations/generate-timeline-summaries?force=true', {
    method: 'POST'
}).then(r => r.json()).then(console.log)
```

### bullets 컬럼이 없다는 에러

원인: DB 마이그레이션 안 됨

해결: 2단계로 돌아가서 bullets 컬럼 추가 SQL 실행

### 타임라인이 표시되지 않는 경우

체크리스트:

1. 이슈가 Track A 이슈인지 확인
```sql
SELECT source_track FROM issues WHERE id = '<issue-uuid>';
```

2. timeline_points에 데이터가 있는지 확인
```sql
SELECT * FROM timeline_points WHERE issue_id = '<issue-uuid>';
```

3. API 응답 확인
```bash
curl http://localhost:3000/api/issues/<issue-uuid>/timeline/summary
```

4. 브라우저 콘솔에서 로그 확인
```
[TimelineSection] Timeline data count: 0
```

## 운영 가이드

### 자동 생성

- update-timeline cron이 매시 30분에 실행
- 새 타임라인 포인트 추가 시 자동 요약 생성

### 수동 생성

- 신규 이슈 급증 시 수동 실행
- 월 1회 전체 재생성 권장 (force=true)

### 성능 최적화

- 배치 크기: 200개 이슈 (더 많으면 여러 번 실행)
- 이슈 간 대기: 800ms (Rate Limit 방지)
- maxDuration: 300초 (5분)
- 예상 처리량: 시간당 약 200개 이슈
