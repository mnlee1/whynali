# 브랜치 구조 가이드

왜난리 프로젝트의 15개 기능별 브랜치 구조 및 관리 방법.

## 브랜치 전략

```
develop (통합 브랜치)
├── 📱 사용자 화면 (3개)
│   ├── feature/user-pages
│   ├── feature/community-discussion
│   └── feature/auth-login
├── 🔧 관리자 화면 (7개)
│   ├── feature/admin-dashboard
│   ├── feature/admin-issues
│   ├── feature/admin-collections
│   ├── feature/admin-votes
│   ├── feature/admin-discussions
│   ├── feature/admin-safety
│   └── feature/admin-logs
└── ⚙️ 백엔드 & 자동화 (5개)
    ├── feature/collectors
    ├── feature/issue-engine
    ├── feature/vote-system
    ├── feature/reactions-comments
    └── feature/safety-bot
```

## 브랜치 작업 흐름

### 1. 새 기능 작업 시작
```bash
# develop 최신 상태로 업데이트
git checkout develop
git pull origin develop

# 작업할 브랜치로 전환
git checkout feature/user-pages
git merge develop
```

### 2. 작업 완료 후
```bash
# 변경사항 커밋
git add .
git commit -m "feat: 이슈 목록 페이지 개선"

# GitHub에 푸시
git push origin feature/user-pages
```

### 3. Pull Request 생성
- GitHub에서 `feature/user-pages` → `develop` PR 생성
- 코드 리뷰 후 머지

### 4. 다른 브랜치에 반영
```bash
# develop 최신화
git checkout develop
git pull origin develop

# 다른 feature 브랜치도 업데이트
git checkout feature/admin-dashboard
git merge develop
```

## 주의사항

### 브랜치 간 의존성
- **공통 컴포넌트**: `components/common/*`, `components/layout/*`는 모든 브랜치에서 공유
- **공통 유틸리티**: `lib/utils/*`, `lib/supabase/*`는 모든 브랜치에서 공유
- **타입 정의**: `types/*`는 모든 브랜치에서 공유

### 충돌 방지
- 같은 파일을 여러 브랜치에서 수정하지 않기
- 공통 파일 수정 시 develop에 먼저 머지 후 다른 브랜치에 반영
- 정기적으로 develop을 각 feature 브랜치에 머지하여 최신 상태 유지

### 커밋 메시지 규칙
```
feat: 새로운 기능 추가
fix: 버그 수정
refactor: 코드 리팩토링
style: 코드 포맷팅
docs: 문서 수정
test: 테스트 추가/수정
chore: 빌드 업무, 패키지 매니저 수정
```
